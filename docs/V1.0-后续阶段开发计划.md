# Agent Memory V1.0 后续阶段开发计划

> 基线：`1.0.0-rc.7`。本计划从 2026-07-21 起衔接阶段 C 验收、生产候选 Gate 和真实 Hermes canary。顺序为“展示身份收尾 → 单 profile canary → 真实数据校准 → 原地晋级 → 稳定化”。

## 1. 执行原则

- `main` 始终保持可部署；开发使用独立分支，只有完整 Gate 通过的提交才能作为生产候选。
- 真实 canary 从初始化起使用最终 project/data/Vault/namespace，通过后原地晋级，不再二次迁库。
- 显示名、布局和星系治理是用户信息架构，不得被当作世界事实或模型推理输入。
- 人工决定优先于后台自动结果；迁移、同步和重建不得覆盖用户改名或映射。
- 每一阶段都必须有可重复测试、审计记录、文档和回滚点；不以“视觉看起来正常”代替数据不变量验证。

## 2. 阶段 E0：Subject 展示身份收尾（canary 前 P0）

目标：恒星只呈现人可读的名称，技术身份保留在详情和审计中，且任何改名都不影响记忆身份。

实施状态（2026-07-21）：E0 代码与迁移已进入候选分支；须在完整 Release Gate、浏览器验收和迁移恢复通过后才可关闭阶段。名称优先级中“读取 Hermes 人格配置”没有稳定且受支持的只读契约，因此 V1 实现为“人工名称 > 可读 `source_profile` > 未命名助手”，不解析 Hermes 私有配置文件。

| ID | 任务 | 完成标准 |
| --- | --- | --- |
| E0-01 | 名称来源数据模型 | 为 Subject 增加 `display_name_origin=default|source|manual`；默认值、约束、迁移和恢复测试完整 |
| E0-02 | 旧数据安全回填 | 已有 `subject.update` 人工名称标为 `manual`；内置 `User`/`Hermes · *` 转换不改变 ID、entity 或 source 映射 |
| E0-03 | 用户恒星显示名 | 用户可在星图中设置名称；未设置时才使用 `User`；禁止自动读取 OS 登录名 |
| E0-04 | Profile 纯人格名 | 自动名称按“人工 > Hermes 配置 > 可读 source > 未命名助手”选择；星图不拼接 `Hermes ·`/`profile`/source key |
| E0-05 | 详情与可访问性 | 详情分开显示“主体类型”、`source_profile` 和实例；视觉标题只有名称；ARIA/tooltip 保留类型语义 |
| E0-06 | 同名与同步保护 | 允许同名 Subject；通过详情稳定标识区分；再次 ingest/profile 同步不覆盖 `manual` 名称 |
| E0-07 | Gate 与主观验收 | 单测、API/DB 集成、迁移/恢复、浏览器、键盘/无障碍、截图对比及完整 Release Gate 通过 |

E0 是真实 canary 前的唯一新功能阻断项。原因是真实 profile 一旦开始长期使用，名称治理和迁移优先级应先稳定，避免之后对真实 Subject 重复回填。

## 3. 阶段 E1：真实 Hermes 单 profile canary（P0）

1. 用户指定一个真实且非关键 profile，同时确认数据范围、模型范围、观察窗口和回滚点。
2. 从已推送且通过 handoff/Release Gate 的提交初始化唯一持久化生产资产；不复用 `/private/tmp` 演练目录。
   初始化、空库验证和备份恢复后必须运行 `production-canary-readiness.sh`，状态固定为 `AWAITING_USER_APPROVAL`。
3. 先在独立 `HERMES_HOME`/TUI 验证 Provider 发现、单 profile 写入、跨会话召回、追溯和 API 断线 fail-soft。
4. 模型默认关闭。本地模型或脱敏外部 API 需单独授权和新生产 key，不复用历史测试 key。
5. 在 2/24/72 小时检查健康、任务失败、证据脱敏、召回越界、恒星名称、星图治理、日志轮转、磁盘增长和备份恢复。

完成标准：真实 profile 产生可追溯 evidence，0 failed job，API 不可用时 Hermes 仍可对话，用户确认恒星、行星、关系和记忆治理可读。

E1 授权边界：readiness 门禁之前不得读取 Hermes profile 列表；门禁之后也不得生成 `hermes-production-*.env`、启动独立 Hermes HOME 或修改任一 Hermes 配置，直到用户确认 profile、数据范围、模型范围、观察窗口和回滚点。

## 4. 阶段 E2：真实数据质量校准（canary 内 P1）

- 以真实日常对话、工具确认和环境观察建立金标样本，分别计算事实准确率、证据可追溯率、重复率和虚假实体率。
- 校准长期/阶段/当前/低价值分类、休眠与唤醒阈值；不为增加显示数量降低证据门槛。
- 增加人际、项目、设备和服务关系样本，观察星系稳定性、重叠成员、桥接关系和单事实 clique 膨胀。
- 验证更正、脱离、忘记、永久清除、实体合并/拆分和 Vault grant/revoke 在真实数据上的派生重建。
- 每个调整批次单独提交、完整 Gate，生产候选只升级到明确验收的 revision。

## 5. 阶段 E3：原地晋级与 V1.0 发布（P1）

- 72 小时 canary、星图主观验收、最新备份恢复和安全检查通过；
- 用户给出可追溯批准参考，执行 `production-promote.sh`；
- 保持同一 database/Vault/namespace/project/image revision，生成 `PROMOTION-RECORD.json`；
- 生成版本 tag、源码包、校验和、镜像/SBOM 清单、升级/回滚说明和最终验收记录；
- 只有备份和用户确认后才退役旧测试/影子容器，不执行全局 Docker prune。

## 6. 阶段 E4：稳定化与 V1 后续（P2）

| 方向 | 范围 |
| --- | --- |
| 多 profile 逐步扩大 | 每次只新增一个 profile，独立验证来源、同名恒星、跨 profile 召回和 Vault 权限 |
| 可观测性 | 健康、任务积压、失败/重试、数据增长、备份年龄、模型费用与脱敏计数 |
| 供应链强化 | 锁定构建工具，显式审核必要的 npm install script，生成 SBOM/签名和可验证发布清单 |
| 灾备与迁机 | 加密备份、Vault key 分离传输、目标主机离线恢复、切换窗口和禁止双写 |
| 通知与主动行为 | 整理报告渠道提醒；状态建议转行为仍需独立权限、渠道策略和审计 |
| V2 边界 | 多用户、多 Agent、云同步、外部知识源和通用 Agent 适配不进入 V1 稳定化补丁 |

## 7. 阶段切换门禁

| 从 | 到 | 必须满足 |
| --- | --- | --- |
| E0 | E1 | 名称迁移可恢复，人工名不被覆盖，浏览器/无障碍/完整 Release Gate PASS |
| E1 | E2 | 指定 profile 产生可追溯证据，fail-soft 和备份恢复 PASS，0 阻断任务失败 |
| E2 | E3 | 完成 72 小时观察、真实数据金标抽检、星图人工验收和最新恢复 Gate |
| E3 | E4 | 晋级记录、tag/发布物、备份与回滚手册完整，用户批准进入长期运行 |

任何方向变更如果会改变 Subject 身份、原始证据不变性、敏感边界、单用户范围或原地晋级原则，必须先向用户报告，不在普通实现批次中默认扩展。
