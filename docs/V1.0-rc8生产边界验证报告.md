## 5. 部署冻结矩阵

使用隔离构建的实际 API/Worker/Migrate image ID 创建只读 deployment bundle：

- 27 个关键运行文件及 bundle 副本 SHA 一致；
- version、Git full revision、镜像 ID、OCI revision 一致时 PASS；
- 替换 API image ID 后按预期失败关闭；
- 修改关键脚本的负例由单元测试验证失败；
- deployment manifest 位于批准 bundle root 之外时拒绝；
- docs 不在关键运行文件集合中，纯文档变更不影响已绑定运行脚本。

## 6. 备份与晋级边界

- `runtime` 验证报告 `pre_canary_ready|pre_canary_only|post_canary`；
- `canary` 遇到 canary 前备份默认失败，仅观察阶段可以显式参数接受；
- `promote` 必须使用 post-canary 恢复验证备份；
- 每个 live source 使用自己的 `first_verified_at`，新增 profile 不能继承旧 profile 的 72 小时；
- 晋级在创建备份前先检查观察窗口，避免过早执行有状态备份；
- 备份失败不会更新 verified backup state，EXIT trap 仍恢复所有已暂停 writer。

## 7. 失败与修复记录

1. 隔离工作树第一次 `uv run pytest` 因宿主 `~/.cache/uv` 沙箱权限失败；改用临时 cache 后发现缺少网络依赖。最终使用主仓库已安装的只读 Python 工具链运行本地回归，并在获准的隔离 Docker 构建中从锁文件重建完整依赖。
2. 原 `release-preflight` 在干净工作树没有 `data/postgres` 时先报“目录不存在”，未先识别主数据树。已调整为路径边界先于存在性检查，回归通过。
3. 设计审查发现第二个 live profile 可能继承第一个 profile 的全局观察时长。已增加每来源 `first_verified_at` 和晋级检查。
4. 初版 deployment manifest 只验证文件与 OCI label，不能发现同 revision label 的镜像被替换。已增加实际 image ID 对比和漂移负例。
5. 初版备份恢复 writer 时忽略 `docker unpause` 失败。已改为显式报错，并增加成功/失败路径契约测试。
