# Agent Memory V1.0 生产候选接入与原地晋级手册

> 适用版本：`1.0.0-rc.7`。本手册是真实 Hermes 和真实数据的生产试运行唯一操作入口。试运行是生产发布阶段，不是临时测试环境；通过后原地晋级，不迁移数据、不更换 Vault、不更换 namespace。

## 1. 不可变生产身份

生产候选初始化时就固定以下资产：

| 资产 | 固定值/规则 |
| --- | --- |
| Compose project | `agent-memory-production` |
| 运行目录 | `$HOME/.local/share/agent-memory/production` |
| 运行配置 | `production.env`，权限 `0600` |
| 数据目录 | `production/postgres`，不使用 Docker 匿名 volume |
| Vault key | `production/vault_root_key`，权限 `0600`，必须单独备份 |
| namespace | `hermes:user-primary` |
| API | `127.0.0.1:7810`，不监听局域网/公网 |
| PostgreSQL | 不发布宿主端口 |
| 状态清单 | `DEPLOYMENT-STATE.json`，不含密钥 |

`canary` 与 `production_active` 是同一套资产的生命周期状态。禁止把已有
`7788/7790/7796–7798` 测试容器、测试数据库或旧 Vault key 改名为生产；也禁止在
canary 通过后新建另一套数据库再复制真实数据。

容器标签只表示 `in-place-canary-to-production` 部署模式，不表示当前
生命周期。当前状态以权限为 `0600` 的 `DEPLOYMENT-STATE.json` 及
晋级后生成的 `PROMOTION-RECORD.json` 为准。

## 2. 真实数据边界

允许进入生产候选的数据：

- 用户指定的真实 Hermes profile 在 canary 期间产生的对话、工具结果和环境观察；
- 用户明确批准的只读 Hermes 历史导出，经过选择计划、脱敏和双 SHA 校验后导入；
- 用户在星图/Vault UI 中手动创建或更正的真实内容。

禁止进入：自动化 fixture、release namespace 数据、影子关系数据库、为了凑数量生成的事实、未授权
profile 的对话、密码/密钥明文。测试只能使用独立 release project/namespace，不能指向生产数据库。

## 3. 初始化最终生产资产

仅在已推送、可交接的干净提交上执行：

```bash
bash scripts/handoff-check.sh
runtime_root="$HOME/.local/share/agent-memory/production"
bash scripts/init-production-env.sh "$runtime_root" "$runtime_root/production.env"
bash scripts/production-up.sh "$runtime_root/production.env"
bash scripts/production-verify.sh "$runtime_root/production.env" empty existing
bash scripts/production-backup.sh "$runtime_root/production.env"
```

首次必须为空库，目的是证明没有测试污染，不是要求 canary 期间保持空库。空库 Gate 通过后，
同一套数据库开始接收真实数据并长期保留。

## 4. 单 profile 真实 canary

用户必须明确指定一个真实 profile。生成配置不会修改或启动 Hermes：

```bash
bash scripts/production-hermes-env.sh \
  "$runtime_root/production.env" \
  '<approved-profile>' \
  PREPARE_HERMES_PRODUCTION_CANARY
```

生成的 `hermes-production-<profile>.env` 使用最终 namespace 和稳定
`source_instance=production-<profile>`。先在独立 `HERMES_HOME`/TUI 验证 Provider 可发现、API 不可用时
fail-soft、无内存工具回灌；再获得用户授权后把配置交给该真实 profile。

真实会话产生后执行：

```bash
bash scripts/production-verify.sh \
  "$runtime_root/production.env" canary existing '<approved-profile>'
```

必须同时看到该 profile 的 source、可追溯 evidence、非零 event 且无 failed job，状态才进入
`canary_active`。

## 5. 观察、模型与人工验收

默认观察窗口为 72 小时，至少在 2/24/72 小时检查：

- API/worker/PostgreSQL 健康、无失败任务、租约可恢复；
- 原始证据可追溯、脱敏正确、跨会话召回不越界；
- 星图实体可读、关系证据充分、人工更正/删除/休眠生效；
- 磁盘增长、日志轮转和备份恢复持续正常；
- 外部模型仍是单独授权面：未完成模型、base URL、数据外发范围和凭据保管审批前，
  model-worker 保持关闭。启用后也必须在真实 canary 期间完成质量与费用验证。

本机或局域网模型使用 `DATA_SCOPE=local`。由于模型 worker 运行在容器内，
宿主机模型必须使用 `host.docker.internal` 而不是 `127.0.0.1/localhost`；也可使用
可达的私网 IP、`.local` 名称或 Docker 内部单段主机名。运行验证会从
model-worker 内执行 DNS/TCP 可达性检查，不发送真实证据。

外部 API 只允许发送经系统脱敏的证据，且必须单独确认：

```bash
# 先把新生产 key 写入临时 0600 文件，不在命令行中传递 key。
bash scripts/production-configure-model.sh \
  "$runtime_root/production.env" \
  '<model-name>' \
  '<https-base-url>/v1' \
  external-redacted \
  ALLOW_REDACTED_PRODUCTION_DATA_TO_MODEL \
  '<key-input-file>'
bash scripts/production-up.sh "$runtime_root/production.env" --existing
bash scripts/production-verify.sh "$runtime_root/production.env" runtime existing
```

脚本把 key 写入 `production/model_api_key` secret，不写入 `production.env`、状态清单、日志或 Git。之前的测试 key 不得复用。

生产晋级不得仅依赖自动计数；用户需在星图和报告中完成人工验收，并给出可追溯的批准编号/记录。

## 6. 原地晋级

晋级命令会重做 canary Gate，生成并恢复验证最新备份，检查观察窗口，然后写入不可混淆的
`PROMOTION-RECORD.json`：

```bash
bash scripts/production-promote.sh \
  "$runtime_root/production.env" \
  PROMOTE_AGENT_MEMORY_PRODUCTION \
  '<approval-reference>' \
  72
```

晋级前后保持同一 Compose project、PostgreSQL 目录、Vault key、namespace、service token 和镜像 revision。
脚本不会复制数据或把测试容器重命名为生产。状态进入 `production_active` 后，同一服务继续运行。

## 7. 停止、回滚与迁机

`production-stop.sh` 只停止容器和网络，不删除数据、Vault 或备份。应用回滚使用已记录的上一镜像；
数据回滚只能从该生产部署的已验证备份恢复，不运行盲目 Alembic downgrade。

跨主机继续开发只需 Git；迁移这套真实生产运行态则必须通过加密通道分开传输：

1. 已验证的 PostgreSQL 备份与 `SHA256SUMS`；
2. `production.env` 和 `DEPLOYMENT-STATE.json`；
3. Vault root key（与数据库备份分开）；
4. Git revision 与对应的发布包/镜像清单。

迁机必须先在目标主机离线恢复验证，再停止旧主机写入并做最终增量/全量备份，禁止两台主机同时使用同一
profile 写入同一逻辑生产空间。
