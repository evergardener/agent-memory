# Agent Memory V1.0 正式迁移与灰度发布方案

> 目标版本：`1.0.0-rc.7`。状态：**源码候选与隔离 Gate 就绪，尚未授权正式迁移、发布或灰度**。

## 1. 发布目标

从已验收源码一次性创建最终生产 project、data、Vault 和 namespace，先对一个真实 Hermes profile 灰度。
现有容器和数据库全部视为测试资产，即使使用过生产数据副本也不直接提升。但新生产候选的真实 canary 数据就是未来生产数据；验收后原地晋级，不再新建数据库。

## 2. 当前事实

- 最近不可变版本为 `1.0.0-rc.6`，不含阶段 C；当前源码已推进到未打 tag 的 `1.0.0-rc.7`；
- 阶段 C 功能提交 `935faf8`，验收记录 `02656de`；
- 当前小样本 Gate 通过，但长期关系质量仍需更多真实会话；
- 正式 namespace 与生产 Hermes Provider 配置未改；
- 根目录 `.env` 缺失仅影响现有测试栈管理，不阻断全新生产候选；现有 test data/Vault/env 不作为生产源；
- 影子环境包含 3 个关系星系、9 个成员和 6 条类型化关系，仅用于验证。
- 独立 release project/data/network/image 前缀、正式关系双 SHA 授权、OCI 溯源和迁移 `0014` 已实现并通过全新隔离环境 Gate。

## 3. 发布前 P0 门禁

下列任一项未通过即 `NO-GO`：

1. **发布隔离**：必须由 `init-release-env.sh` 创建独立 project、镜像前缀、data/backup/Vault 路径、
   namespace、端口和网段；`release-preflight.sh` 必须通过；
2. **生产候选隔离**：必须由 `init-production-env.sh` 创建最终 `agent-memory-production`/data/Vault/`hermes:user-primary`，并通过 host、空数据、安全属性、迁移和备份恢复 Gate；
3. **生产关系来源**：不得 DB COPY 测试关系或 galaxy 行。必须使用已审关系 preview 的计划 SHA、正式
   namespace 原文、固定确认短语、change ID、已恢复验证备份的 `SHA256SUMS` 及其 SHA 执行；
4. **备份闭环**：发布前新备份、SHA 校验、空库恢复、Vault 实际解密全部通过，根密钥另存；
5. **安全门禁**：evidence 不变、裸 provider key 金丝雀为 0、Vault 明文不进入 API/星图/日志/模型；
6. **版本一致**：`VERSION`、`pyproject.toml`、Compose 默认标签、`.env`、CHANGELOG 和 Release 说明一致；
7. **用户授权**：明确授权正式 namespace 迁移、单 profile Hermes 灰度和候选版本发布，三项可分别批准。

## 4. 发布阶段

### 阶段 R0：冻结与记录

- 记录 `git status`、HEAD、运行镜像 ID、Compose 状态、数据库计数和当前 Hermes Provider；
- 停止非必要历史导入与模型评测，保持生产 Hermes 原始数据只读；
- 创建发布工单/审计 ID，所有备份、计划 SHA、镜像和验收结果关联到同一 ID。

### 阶段 R1：隔离构建与全量 Gate

- 在独立数据目录、独立 Docker project 和独立 namespace 构建 `rc.7`；
- 运行 Ruff、完整 pytest、前端 typecheck/build、迁移、API/Hermes 回归、worker outage/recovery、
  backup/restore/Vault decrypt；
- 运行脱敏固定点检查和 provider-key 金丝雀；
- 输出不可变镜像 ID、架构、Git revision 和 Gate JSON。失败时不进入正式数据库。

```bash
runtime_root="$(mktemp -d /private/tmp/agent-memory-release-gate.XXXXXX)"
AGENT_MEMORY_RELEASE_PROJECT=agent-memory-release-rc7 \
  bash scripts/init-release-env.sh "$runtime_root" "$runtime_root/release.env"
bash scripts/release-preflight.sh "$runtime_root/release.env"
bash scripts/release-check.sh "$runtime_root/release.env"
```

`release-check.sh` 必须从干净 Git 工作树运行，并核对镜像 OCI version/revision；不得传入生产 `.env`。

### 阶段 R2：创建最终生产候选

- 使用生产候选手册创建全新空数据库与最终 Vault key、`hermes:user-primary` 和容器；
- 验证 `0001 -> 0014` 空库迁移、OCI revision、安全属性和 0 evidence/fact/Vault/failed job；
- 完成生产候选数据库 dump、校验和、临时空库恢复和 Vault 解密检查；
- 不导入当前测试数据库；需要的真实历史数据后续从只读原始导出重新选择和脱敏。

### 阶段 R3：真实单 profile canary

1. 用户选择一个非关键 profile；其他 profile 不变；
2. 先为独立 Hermes HOME/TUI 生成 canary env，不自动安装或修改 Hermes；
3. 授权后执行普通偏好、工具确认事实、跨会话召回和 API 离线 fail-soft；
4. 验证该 profile source、evidence、任务、脱敏、追溯与备份；
5. 观察 2/24/72 小时；失败时停止 canary，生产候选数据保留用于复盘和重试。

正式关系先预览，不写数据库：

```bash
uv run agent-memory-apply-reviewed-relations \
  --source '<normalized-backup.jsonl>' \
  --selection '<selection.json>' \
  --gold '<reviewed-gold.json>' \
  --namespace 'hermes:user-primary'
```

人工核对 preview 后，执行时必须同时提供：

```bash
uv run agent-memory-apply-reviewed-relations \
  --source '<normalized-backup.jsonl>' \
  --selection '<selection.json>' \
  --gold '<reviewed-gold.json>' \
  --namespace 'hermes:user-primary' \
  --apply --confirm-plan-sha256 '<plan-sha256>' \
  --allow-production \
  --confirm-production-namespace 'hermes:user-primary' \
  --confirm-production-apply 'APPLY_REVIEWED_RELATIONS_TO_PRODUCTION' \
  --backup-manifest '<verified-backup>/SHA256SUMS' \
  --confirm-backup-manifest-sha256 '<manifest-sha256>' \
  --change-id '<release-change-id>'
```

### 阶段 R4：原地晋级

- 72 小时 canary、星图人工验收、最新备份恢复和用户授权全部通过；
- 执行 `production-promote.sh`，写入批准参考、镜像 ID、revision 和备份清单的 `PROMOTION-RECORD.json`；
- 保持同一 project、数据目录、Vault key、namespace、service token 和镜像，不迁移/复制真实数据；
- 后续 profile 扩大仍逐个授权，不因状态改名自动全量接入。

### 阶段 R5：扩大与发布固化

- 灰度无阻断问题后，再逐 profile 启用；
- 更新 Release 说明和验收矩阵，打 tag，生成源码包、镜像记录和 SHA256SUMS；
- 正式候选稳定后，备份影子库并经用户确认清理 `7796`、`7797`、`7798` 临时容器；
- `7790` 导入栈按是否仍需历史导入单独处理，不和影子容器一起删除。

## 5. 灰度验收指标

| 维度 | 必须满足 |
| --- | --- |
| 可用性 | API ready；Hermes 在 API 离线时 fail-soft；无持续失败任务 |
| 数据完整性 | 原始 evidence 哈希不变；turn/event/source 可追溯；重复写入幂等 |
| 召回质量 | 明确主题可召回历史细节；普通召回不唤醒 forgotten/isolated/purged |
| 星系质量 | 天体仅 Subject/实体；无内部 ID/句子节点；成员与关系均有 evidence |
| 治理 | 编辑、隐藏、固定、排除、撤销和布局重建后保留；并发冲突失败关闭 |
| 安全 | Vault 明文、API key、密码、地址等不进入星图、日志、索引和模型输入 |
| 运维 | 备份恢复与 Vault 解密通过；日志无阻断错误；资源增长可解释 |

## 6. 回滚策略

优先顺序：

1. **Hermes 侧止血**：灰度 profile 切回原 Provider；Agent Memory 故障不得阻断主回合；
2. **应用回滚**：停止新 API/worker，使用该部署已记录的上一镜像 ID恢复；不把当前 rc.6 测试容器当作生产回滚目标，也不盲目 Alembic downgrade；
3. **数据回滚**：仅在新写入或迁移破坏数据时，停写后恢复发布前 dump 与匹配的 Vault 根密钥；
4. **关系层回滚**：隔离/休眠新关系与星系投影，保留原始 evidence 和审计，不删除故障现场；
5. **复盘**：保存日志、任务 ID、计划 SHA、镜像 ID和计数差异，再决定修复或重新发布。

回滚后必须验证 API、旧 Provider、证据计数、Vault 解密和备份可恢复性。禁止以清库、全局 Docker prune
或删除备份作为回滚手段。
