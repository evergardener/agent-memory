# Agent Memory V1.0 正式迁移与灰度发布方案

> 建议目标版本：`1.0.0-rc.7`。状态：**方案就绪，尚未授权执行**。

## 1. 发布目标

把阶段 C 的关系星系能力从已验收影子环境安全提升为新的不可变候选版本，并只对一个 Hermes profile
灰度。发布不复制影子数据库行，不修改 Hermes 原始会话，不把 API 暴露到 localhost 以外，也不把
外部模型设为上线前置条件。

## 2. 当前事实

- 最近不可变版本为 `1.0.0-rc.6`，不含阶段 C；
- 阶段 C 功能提交 `935faf8`，验收记录 `02656de`；
- 当前小样本 Gate 通过，但长期关系质量仍需更多真实会话；
- 正式 namespace 与生产 Hermes Provider 配置未改；
- 根目录 `.env` 缺失，但在线标准容器、数据库目录、Vault 根密钥和备份仍存在；
- 影子环境包含 3 个关系星系、9 个成员和 6 条类型化关系，仅用于验证。

## 3. 发布前 P0 门禁

下列任一项未通过即 `NO-GO`：

1. **恢复管理面**：从校验通过、与在线数据库匹配的备份恢复 `.env`，验证 Compose 配置与 `7788`；
2. **发布隔离**：完整 release-check 不得复用在线 `data/postgres`。当前 Compose 固定数据目录和 IPAM，
   需先支持独立数据根目录、独立 Compose project 和不冲突网段，或安排明确停机窗口在完全隔离副本执行；
3. **生产关系来源**：不得 DB COPY 影子 `entity_relations`/galaxy 行。当前已审关系写入器故意拒绝正式
   namespace；发布前必须实现并评审“正式 namespace 确认计划 + SHA + 审计”的提升路径，或确认正式
   数据可以由确定性在线流程完整重建；
4. **备份闭环**：发布前新备份、SHA 校验、空库恢复、Vault 实际解密全部通过，根密钥另存；
5. **安全门禁**：evidence 不变、裸 provider key 金丝雀为 0、Vault 明文不进入 API/星图/日志/模型；
6. **版本一致**：`VERSION`、`pyproject.toml`、Compose 默认标签、`.env`、CHANGELOG 和 Release 说明一致；
7. **用户授权**：明确授权正式 namespace 迁移、单 profile Hermes 灰度和候选版本发布，三项可分别批准。

## 4. 发布阶段

### 阶段 R0：冻结与记录

- 记录 `git status`、HEAD、运行镜像 ID、Compose 状态、数据库计数和当前 Hermes Provider；
- 停止非必要历史导入与模型评测，保持生产 Hermes 原始数据只读；
- 创建发布工单/审计 ID，所有备份、计划 SHA、镜像和验收结果关联到同一 ID。

### 阶段 R1：隔离构建与全量 Gate

- 在独立数据目录、独立 Docker project 和独立 namespace 构建 `rc.7`；
- 运行 Ruff、完整 pytest、前端 typecheck/build、迁移、API/Hermes 回归、worker outage/recovery、
  backup/restore/Vault decrypt；
- 运行脱敏固定点检查和 provider-key 金丝雀；
- 输出不可变镜像 ID、架构、Git revision 和 Gate JSON。失败时不进入正式数据库。

### 阶段 R2：正式数据库预演

- 对最新生产备份恢复副本执行 `0001 -> 0013` 升级；
- 比较升级前后 evidence 全字段哈希和核心计数；
- 在副本中运行正式关系提升计划和社区重建，检查 0 单体星系、全部成员证据可追溯、人工治理不被覆盖；
- 使用旧版本 API 对升级后副本做兼容性探测，确定应用回滚是否可保留新表。

### 阶段 R3：维护窗口部署

1. 暂停 Agent Memory 写入，不停止 Hermes 本身；
2. 创建并验证最终备份，另存 Vault 根密钥；
3. 拉起 `migrate`，必须成功退出；
4. 启动 core worker 和 API，先保持 model-worker 关闭；
5. 验证 health、认证、namespace、召回、证据追溯、Vault 脱敏标记和星图；
6. 运行关系提升计划和 `rebuild_communities`，记录计划 SHA、任务 ID、输入快照哈希和输出计数；
7. 未通过则立即停止新写入并进入回滚。

### 阶段 R4：单 profile 灰度

- 只选择一个非关键 Hermes profile，其他 profile 保持原 Provider；
- 先验证 Provider 发现、turn sync、prefetch、显式 recall、trace、更正、Vault grant 和 API 离线 fail-soft；
- 观察至少 72 小时，分别在 2、24、72 小时生成检查记录；
- 样本不足时可以延长观察，但不能通过降低证据和社区门槛制造星系。

### 阶段 R5：扩大与发布固化

- 灰度无阻断问题后，再逐 profile 启用；
- 更新 Release 说明和验收矩阵，打 tag，生成源码包、镜像记录和 SHA256SUMS；
- 正式候选稳定后，备份影子库并经用户确认清理 `7796`、`7797`、`7798` 临时容器；
- `7790` 导入栈按是否仍需历史导入单独处理，不和影子容器一起删除。

## 5. 灰度验收指标

| 维度 | 必须满足 |
| --- | --- |
| 可用性 | API ready；Hermes 在 API 离线时 fail-soft；无持续失败任务 |
| 数据完整性 | 原始 evidence 哈希不变；turn/event/source 可追溯；重复写入幂等 |
| 召回质量 | 明确主题可召回历史细节；普通召回不唤醒 forgotten/isolated/purged |
| 星系质量 | 天体仅 Subject/实体；无内部 ID/句子节点；成员与关系均有 evidence |
| 治理 | 编辑、隐藏、固定、排除、撤销和布局重建后保留；并发冲突失败关闭 |
| 安全 | Vault 明文、API key、密码、地址等不进入星图、日志、索引和模型输入 |
| 运维 | 备份恢复与 Vault 解密通过；日志无阻断错误；资源增长可解释 |

## 6. 回滚策略

优先顺序：

1. **Hermes 侧止血**：灰度 profile 切回原 Provider；Agent Memory 故障不得阻断主回合；
2. **应用回滚**：停止 rc.7 API/worker，使用已记录镜像 ID恢复 rc.6；优先保留向前兼容的新表，不盲目
   Alembic downgrade；
3. **数据回滚**：仅在新写入或迁移破坏数据时，停写后恢复发布前 dump 与匹配的 Vault 根密钥；
4. **关系层回滚**：隔离/休眠新关系与星系投影，保留原始 evidence 和审计，不删除故障现场；
5. **复盘**：保存日志、任务 ID、计划 SHA、镜像 ID和计数差异，再决定修复或重新发布。

回滚后必须验证 API、旧 Provider、证据计数、Vault 解密和备份可恢复性。禁止以清库、全局 Docker prune
或删除备份作为回滚手段。

## 7. 发布产物

- Git tag 和完整 revision；
- `agent-memory-1.0.0-rc.7-source.tar.gz`；
- API/worker/migrate 镜像 ID与架构；
- `SHA256SUMS`、Compose、`.env.example`、迁移清单；
- Release Gate JSON、恢复验证结果、Hermes 灰度记录；
- 脱敏/金丝雀结果、正式关系计划 SHA、社区输入快照哈希；
- 已知限制、回滚点、Vault 根密钥保管确认（不包含密钥本身）。

## 8. 当前结论

当前可以继续做发布准备，但不能直接上线。最先应完成的是 `.env` 管理面恢复、并行发布栈隔离和正式
关系提升路径；完成这三项后再申请正式迁移与单 profile 灰度授权。
