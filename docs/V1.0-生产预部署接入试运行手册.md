# Agent Memory V1.0 生产预部署接入试运行手册

> 适用版本：`1.0.0-rc.7`。本文覆盖从全新生产预部署容器创建，到下一阶段可以为一个 Hermes profile 准备 canary 接入配置为止。任何命令均不得复用当前 `7788/7790/7796–7798` 测试容器、测试数据库或测试 Vault key。

## 1. 预部署边界

预部署不是正式生产切换：

- 使用全新 Compose project、镜像前缀、PostgreSQL 数据目录、Vault key、备份目录和 namespace；
- API 只绑定 `127.0.0.1:7810`，PostgreSQL 不发布宿主端口；
- backend/edge 默认使用 `172.16.250.0/24`、`172.16.251.0/24`，与当前 Docker 网络和局域网 `192.168.7.0/24` 不重叠；
- model-worker、import 服务和外部模型默认不创建、不启用，也不保存模型 key；
- 初始数据库必须为空，不从现有测试库复制关系、星系、任务或测试生成数据；
- 初始化和启动不会安装 Hermes 插件、修改 Hermes profile、启动 Hermes TUI 或写入正式 namespace；
- 只有用户明确选择非关键 profile 后，才生成一份独立的 canary 环境文件；生成文件本身仍不修改 Hermes。

现有测试容器使用过真实对话副本，但仍属于可退役验证资产，不是正式部署源。需要保留的历史对话应从只读原始导出重新走选择计划、脱敏、双 SHA 和导入流程，不能把测试数据库整体当成生产种子。

## 2. 创建全新预部署运行目录

建议把运行资产放在非 Git、非 iCloud 项目目录：

```bash
runtime_root="$HOME/.local/share/agent-memory/predeploy-1.0.0-rc.7"
bash scripts/init-predeploy-env.sh "$runtime_root" "$runtime_root/predeploy.env"
```

脚本创建：

- `predeploy.env`：`0600`，含随机数据库密码、service token 和 UI hash；
- `vault_root_key`：`0600`，与测试 Vault key 完全独立；
- `postgres/`：全新的空数据目录；
- `backups/`：只保存这套预部署的备份；
- 星图登录密码只显示一次，不写入 env、状态文件或 Git。

初始化后尚未创建容器，也未接入 Hermes。

## 3. 启动并完成空栈 Gate

必须从干净 Git 提交执行：

```bash
bash scripts/predeploy-up.sh "$runtime_root/predeploy.env"
```

启动脚本依次验证：

1. 版本、完整 Git revision、路径、密钥权限、namespace、端口、网段和模型关闭状态；
2. 当前 Docker 网络不存在重叠，`7810` 未占用，同名 Compose project 不存在；
3. 构建 API/worker/migrate 镜像，OCI version/revision 必须等于 env 和 HEAD；
4. 只启动 `postgres`、`migrate`、`api`、`worker`；
5. API readiness、未认证 401、service token 认证、namespace/version；
6. migration 位于当前 Alembic head；
7. API/worker 只读根文件系统、`cap_drop=ALL`、`no-new-privileges` 与网络隔离；
8. evidence、fact、Vault entry、failed job 均为 0；
9. 写入不含秘密的 `PREDEPLOY-STATE.json`，状态为 `ready_for_canary`。

重复启动已有预部署时使用：

```bash
bash scripts/predeploy-up.sh "$runtime_root/predeploy.env" --existing
```

它会保留数据和 canary 状态，不允许偷偷切换版本、revision、namespace 或 project。

## 4. 运行检查与备份闭环

只读检查：

```bash
bash scripts/predeploy-verify.sh \
  "$runtime_root/predeploy.env" runtime existing
```

在首次接入 Hermes 之前，必须完成一次预部署备份和空库恢复：

```bash
backup_dir="$(bash scripts/predeploy-backup.sh "$runtime_root/predeploy.env")"
```

备份包含数据库 dump、运行 env、基础/预部署 Compose、状态文件、版本、锁文件和 `SHA256SUMS`；脚本会自动恢复到临时空库、比较计数并运行 Vault 解密检查。Vault root key 仍需单独复制到安全介质，不能只和数据库备份放在一起。

## 5. 为下一阶段准备单 profile canary

本阶段只准备配置，不启动或修改 Hermes：

```bash
bash scripts/predeploy-hermes-env.sh \
  "$runtime_root/predeploy.env" \
  '<non-critical-profile>' \
  PREPARE_HERMES_PREDEPLOY_CANARY
```

生成的 `hermes-canary-<profile>.env` 权限为 `0600`，只包含 loopback API、service token、预部署 namespace、明确 profile 和 instance。它不包含数据库密码、UI hash或 Vault key。`PREDEPLOY-STATE.json` 变为 `canary_config_prepared`，但 `hermes_connected` 仍为 `false`。

下一阶段进入试运行时，先在独立 `HERMES_HOME`/TUI 验证插件安装和 fail-soft；获得用户授权后才能把这份环境配置交给选定 profile。至少完成三类对话：普通偏好、工具确认事实、跨会话显式召回，然后运行：

```bash
bash scripts/predeploy-verify.sh \
  "$runtime_root/predeploy.env" canary existing '<non-critical-profile>'
```

只有数据库中出现该 profile 的 source 和可追溯 evidence、API/容器/迁移/安全检查继续通过时，状态才更新为 `canary_active`。

## 6. 观察与停止

试运行观察点：2 小时、24 小时、72 小时。每次检查 API readiness、failed jobs、证据追溯、脱敏、普通/显式召回差异、星系可读性和磁盘增长。模型保持关闭，样本不足不能通过降低关系门槛制造星系。

无损停止：

```bash
bash scripts/predeploy-stop.sh "$runtime_root/predeploy.env"
```

该命令只删除预部署容器和网络，保留 PostgreSQL 数据、备份、Vault key、env 和状态；随后可用 `predeploy-up.sh --existing` 恢复。禁止用 `docker system prune`、删除 runtime root 或重建 Vault key 作为回滚。

## 7. 进入试运行的准入条件

- 仓库干净且提交已推送；
- `predeploy-up`、`predeploy-verify empty`、`predeploy-backup`/restore 全部通过；
- `PREDEPLOY-STATE.json` 为 `ready_for_canary`；
- 预部署 API 只在 `127.0.0.1:7810`，model-worker 未运行；
- 当前测试容器未被重启或当作生产源；
- 用户明确给出 canary profile 和接入授权。

达到以上条件后，下一阶段可以直接进入“独立 Hermes → 单 profile → 2/24/72 小时”的生产预部署试运行，不再需要修改部署代码。
