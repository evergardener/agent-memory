# Agent Memory V1.0 开发部署与运维手册

> 更新日期：2026-07-22。本文描述当前仓库实际运行方式；发布决策以
> [`V1.0-正式迁移与灰度发布方案.md`](V1.0-正式迁移与灰度发布方案.md) 为准。

## 1. 适用范围与安全边界

V1 是单用户、本地优先、仅接入 Hermes 的 Docker 服务。API/UI 只绑定宿主 `127.0.0.1`，PostgreSQL
不发布端口。除显式启用 model-worker 外，数据不离开本机。以下操作默认只管理本项目，不授权修改生产
Hermes 会话、profile 配置或其他容器。

必须遵守：

- 原始 evidence 只读；更正、忘记、隔离和清除通过派生状态与审计实现；
- `.env`、`secrets/`、`data/`、`backups/` 和 `release-artifacts/` 不进入 Git；
- Vault 根密钥与数据库备份分开保存；丢失根密钥后密文不可恢复；
- 任何历史导入、模型评测和社区生成先进入 staging/shadow namespace；
- 不把 `7788`、PostgreSQL 或 Vault 接口暴露到局域网、公网或反向代理。

## 2. 运行拓扑

| 组件 | 职责 | 网络/暴露 |
| --- | --- | --- |
| `postgres` | 证据、记忆、关系、星系、审计、状态和 Vault 密文 | 仅 `backend`，无宿主端口 |
| `migrate` | 一次性执行 Alembic 升级 | 仅 `backend`，成功退出后 API/worker 才启动 |
| `api` | API、星图 UI、登录和治理 | `backend + edge`，默认 `127.0.0.1:7788` |
| `worker` | 确定性抽取、索引、生命周期、报告和社区重建 | 仅 `backend` |
| `model-worker` | 可选模型增强 | `backend + edge`；只有它接收模型 key |
| `import-*` | 历史导入 staging | profile `import`，默认 UI/API `127.0.0.1:7790` |

固定 Docker 网段为 `172.16.240.0/24` 和 `172.16.241.0/24`，与本地
`192.168.7.0/24` 局域网不重叠。换主机前仍须检查宿主、VPN 和现有 Docker 地址池；发生重叠时先修改
部署设计，不要直接启动。正式 API 只通过 `edge` 获得宿主端口，数据库和核心 worker 保持 internal 网络。

## 4. 新安装

只在全新目录、没有既有数据库和密钥时执行：

```bash
bash scripts/init-local.sh
docker compose --env-file .env config --quiet
docker compose --env-file .env up -d --build
docker compose --env-file .env ps
curl --fail http://127.0.0.1:7788/health/ready
```

初始化脚本只显示一次星图密码，并生成数据库密码、服务 token、UI 会话密钥和 Vault 根密钥。保存密码后，
立即把 `secrets/vault_root_key` 复制到独立安全介质。

## 5. 当前测试栈的 `.env` 恢复（非上线前置）

当前目录不是新安装。不得运行 `scripts/init-local.sh`，否则会生成与在线数据库不匹配的新数据库密码和
服务 token。优先从最近、校验通过且与当前数据库同一时期的备份恢复：

```bash
cd backups/20260719T043257Z
shasum -a 256 -c SHA256SUMS
cd ../..
install -m 600 backups/20260719T043257Z/runtime.env .env
docker compose --env-file .env config --quiet
```

这只用于需要继续管理或备份当前测试栈的场景，不是生产候选初始化步骤。它也是当前最晚的候选备份，不代表可盲目使用。恢复前需比较备份时间、`VERSION`、运行容器创建时间和
Vault 根密钥来源；不得打印 `runtime.env`，不得把它加入 Git。若无法证明匹配，保持现有容器运行并停止
重建，先做新的容器/数据库只读审计。

## 6. 开发环境与回归

```bash
uv sync --frozen --extra dev --extra migrations
uv run ruff check src integrations tests migrations
uv run pytest -q
npm --prefix frontend ci
npm --prefix frontend run typecheck
npm --prefix frontend run build
```

普通开发测试不应写 `hermes:user-primary`。API/Provider/故障回归统一使用
`hermes:automated-tests*` 与 `scripts/verify-isolated-regression.sh`。历史真实数据只允许使用已脱敏副本，
并与合成故障测试分开评价。

开发服务启动前先检查网段和端口，再使用独立 namespace。发布级并行测试必须由
`scripts/init-release-env.sh` 生成独立 project、镜像前缀、data/backup/Vault 路径、端口和网段；不要复制
标准 `.env`，也不要把生产 `.env` 传给 release-check。

生产试运行直接创建最终生产资产，canary 通过后原地晋级：

```bash
runtime_root="$HOME/.local/share/agent-memory/production"
bash scripts/init-production-env.sh "$runtime_root" "$runtime_root/production.env"
bash scripts/production-up.sh "$runtime_root/production.env"
bash scripts/production-backup.sh "$runtime_root/production.env"
```

它不会复用测试 data/Vault/env，首次启动不启动 model-worker，也不自动修改 Hermes。完整操作见
[`V1.0-生产候选接入与原地晋级手册.md`](V1.0-生产候选接入与原地晋级手册.md)。

每个开发批次推送后运行 `scripts/handoff-check.sh`；它是“另一台主机可直接接手”的最低门禁。

## 7. 日常启动、停止与升级

```bash
docker compose --env-file .env config --quiet
docker compose --env-file .env up -d
docker compose --env-file .env ps
docker compose --env-file .env logs --since 15m migrate api worker model-worker
curl --fail http://127.0.0.1:7788/health/ready
```

停止应用但保留数据库：

```bash
docker compose --env-file .env stop api worker model-worker import-api import-worker import-model-worker
```

升级前必须先备份并验证恢复。不要修改已执行的迁移文件，不要手工修改 Alembic 版本。`migrate` 非零时
停止升级，检查日志；不要让 API 绕过迁移依赖启动。

## 8. 备份与恢复

```bash
backup_dir="$(bash scripts/backup.sh .env)"
cp secrets/vault_root_key "$backup_dir/vault_root_key.separate-copy"
bash scripts/verify-restore.sh "$backup_dir" .env
```

实际长期保存时，`vault_root_key.separate-copy` 应移到数据库备份之外的受控介质；上面的复制仅用于提醒
必须形成对应副本。恢复验证会创建临时数据库、比较核心表计数并实际解密 Vault，然后清理临时库。

恢复生产数据库的破坏性步骤不在自动脚本中，必须在停写、保留故障现场和明确授权后单独执行。

## 9. Hermes 接入与停用

服务健康后安装托管 Provider：

```bash
python3 scripts/hermes-plugin.py install --hermes-home "${HERMES_HOME:-$HOME/.hermes}"
hermes memory setup agent_memory
```

Provider 只访问本地 API。多 profile 共享 `AGENT_MEMORY_NAMESPACE`，各自保留 `source_profile`；不同 profile 投影为不同人格恒星，同 profile 多实例复用同一恒星。灰度按 profile 逐个批准，每个 `live_profile` 必须用自己的 `source_instance`、event 和观察窗口通过 Gate；历史导入登记为 `historical_import`，不能替代 live gate。出现异常先把对应 profile 切回原 Provider，再处理 Agent Memory，避免阻断 Hermes 主回合。卸载脚本只删除带 `.agent-memory-managed` 标记的目录。

生产来源、运行冻结和备份新鲜度命令见 [`V1.0-生产来源治理与部署冻结设计.md`](V1.0-生产来源治理与部署冻结设计.md)。源码或关键脚本改变后，旧运行态操作应失败关闭；不得绕过 deployment manifest，也不得在用户确认前 rebuild/restart。

## 10. 模型启用

默认 `AGENT_MEMORY_MODEL_ENABLED=false`。启用前确认供应商、base URL、模型、费用、数据授权和超时；
API key 只写 `.env` 并只注入 model-worker。外部数据发送必须显式设置
`AGENT_MEMORY_MODEL_ALLOW_EXTERNAL_DATA=true`，否则失败关闭。日志、报告和 Git 差异中不得出现 key。

## 11. 观测与排障顺序

1. `docker compose ... ps`：确认 postgres health、migrate exit 0、API health；
2. `/health/live` 与 `/health/ready`：区分进程存活和依赖就绪；
3. `logs --since 15m`：只查看当前故障窗口，区分警告与阻断错误；
4. worker 积压：查看 `ops.jobs.last_error_code`，保留失败任务，不直接删除；
5. Hermes 无记忆：检查 URL、token、namespace、source profile 和 worker，而不是修改原始会话；
6. Vault 解密失败：核对数据库与根密钥时间点，禁止生成新 key 覆盖；
7. 网路异常：比较宿主与临时容器到目标地址的可达性，检查 IPAM/VPN/LAN 冲突。

## 12. 临时阶段 C 容器

`agent-memory-phase-c-api`、`agent-memory-phase-c-ui` 和 `agent-memory-phase-c-shadow-api` 是验收资产，
不是正式 Compose 发布物。正式迁移完成、影子库备份验证通过且用户明确同意前不要删除。清理时按容器、
镜像、网络和数据库分别列清单，禁止使用全局 Docker prune。
