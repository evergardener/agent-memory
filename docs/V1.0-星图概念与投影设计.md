# Agent Memory for Hermes — V1.0 星图概念与投影设计

## 1. 文档状态与目标

状态：**阶段 A/B 已在 `1.0.0-rc.6` 实现并验证；阶段 C 与阶段 D 工程验证已完成，并在当前小样本范围内验收通过；2026-07-21 冻结的恒星展示身份规则待阶段 E0 实现，正式 canary 仍需单独授权**。

本文重新定义星图的空间语义，解决把“长期、阶段、当前、情节”直接作为星系后，普通实体难以稳定归属、同一实体被迫跨类别重复出现的问题。本文是 [`V1.0-项目需求文档.md`](V1.0-项目需求文档.md) 的星图专项设计；`1.0.0-rc.6` 只包含主体/行星与观察投影，关系社区和布局治理位于其后的阶段 C 提交，不能反向视为 rc.6 发布内容。

本次调整只改变主体身份建模、图谱投影、布局和导航，不改变下列基础约束：

- 原始证据保持只读；
- 事实、情节和长期脉络继续可追溯、可重建；
- 长期、阶段、当前继续作为事实分类和召回/lifecycle 依据；
- Vault 明文继续与普通记忆、检索、模型和星图隔离；
- 事实、摘要、状态值和自动生成句子不得成为星体。

## 2. 核心判断

时间分类描述“记忆处于什么阶段”，实体关系描述“对象与谁、与什么主题相连”。两者是正交维度，不能共用一套星系边界。

因此：

1. 星图空间由**主体与实体关系拓扑**决定；
2. 长期、阶段、当前、休眠、忘记等成为**观察镜片**；
3. 情节和长期脉络成为**可播放的关系投影**；
4. 星系是可重建、可重叠、可人工治理的实体社区，不是事实分类的存储位置。

## 3. 统一天体语义

| 星图概念 | 系统语义 | 可持久化对象 | 不得表达 |
| --- | --- | --- | --- |
| 恒星 | 拥有第一人称视角、可产生或接收记忆的主体 | 用户主体、Hermes profile 人格 | 普通人物、项目、服务、句子或事实 |
| 行星 | 唯一规范实体 | 人物、项目、设备、服务、地点、组织、工具、技术等 | 事实片段、状态键、来源 ID、内部任务 |
| 星系 | 关系紧密的行星社区 | 可重建的社区投影、人工固定的社区边界 | 长期/阶段/当前等事实类别 |
| 星座 | 某个情节被激活时形成的临时关系图形 | episode 及其实体/事实投影 | 永久实体身份或无证据叙事 |
| 星流/轨迹 | 多个情节形成的长期演化路径 | arc 及其 episode/fact 投影 | 新事实或因果推断 |
| 光线/连线 | 有证据支持的实体关系及强度 | 实体关系、支撑事实、人工关联 | 仅因布局接近而推断的关系 |
| 脉冲/光晕 | 当前观察镜片下的状态 | 活跃度、事实类型、生命周期、敏感标记 | 新节点 |
| 星尘/注释 | 事实、摘要、来源与治理提示 | 脱敏事实投影和来源引用 | 可点击实体替身 |

### 3.1 恒星不是 `person` 的别名

恒星表达系统中的“记忆主体角色”，不是现实世界中的生物类别：

- 本地用户是一个稳定恒星主体；
- 每个被配置为独立人格的 Hermes profile 是一个恒星主体；
- 同一 profile 的多个 TUI、实例和会话只映射到同一恒星，不新增恒星；
- 其他人物仍是普通 `person` 行星；只有未来被显式接入为系统主体时，才通过可审计映射成为恒星投影；
- 主体仍可引用一个规范实体记录，以参与召回和关系图；同一规范实体在同一画面中不得同时显示为恒星和行星。

### 3.2 主体恒星群

主宇宙中心不再固定为写死的 User/Hermes 双星，而是“主体恒星群”：

- 用户恒星固定在中心；
- 已配置的 profile 人格恒星以稳定位置分布在用户周围；
- 当前访问页面或产生记忆的 profile 高亮，其他 profile 降低亮度但不消失；
- profile 改名只更新显示名，不改变稳定主体 ID；
- 用户恒星使用用户明确设置的显示名；未设置时可临时显示 `User`，但不得自动读取操作系统登录名；
- profile 恒星标题只显示人格名称，系统不自动拼接 `Hermes ·`、`profile`、括号中的 source key 或实例信息；
- 名称优先级为“人工显示名 > Hermes 人格/配置名 > 经脱敏且可读的 source profile 名 > 未命名助手”，自动同步不得覆盖人工名称；
- UUID、时间戳、实例 ID、`derived-*` 或其他内部标识不得作为恒星可见名称；
- 详情页单独标注“用户”或“Hermes Profile”主体类型，并将 `source_profile`/实例放入技术来源区；视觉标题不重复类型字样；
- 两个 profile 可使用相同显示名，不为消除歧义而改写名称；详情、tooltip 与无障碍标签使用稳定类型/来源语义区分。

V1 的共享 namespace 仍是统一记忆边界。恒星区分视角和来源，不形成新的数据库或权限租户。

连续互动状态可以按 profile 人格恒星分轨显示并继承 namespace 默认配置；这种分轨只表示人格的当前互动节奏，不拆分事实、实体、情节或长期脉络。

## 4. 行星与实体身份

每个行星对应一个 canonical entity：

- 一个实体全宇宙只有一个稳定 ID；
- 同义名、旧名称和来源别名映射到同一实体；
- 行星可以同时参加多个星系和星座，但不能复制为多个实体；
- 实体合并必须保留可撤销映射；拆分只迁移用户选定的事实关联；
- 关系不足的实体进入“深空/未成系”区域，不为满足布局强制归类；
- 单个实体可形成可聚焦的“行星详情”，但不得自动生成只有一个行星的星系。

规范化先于社区计算。产品名、旧名称、UI 名称和运行时别名必须先合并，例如 `Hermes WebUI` 与
`Hermes Studio` 统一映射为可读实体 `Hermes WebUI / Studio`；别名共同出现只增加实体证据，不能
生成实体到自身的关系。

应用内部模块默认不是行星。类似 `Agent Bridge` 的对象只有同时具备独立配置、独立生命周期、独立
治理入口和用户可识别身份时，才可提升为服务行星；否则它只是应用关系的实现组件或边属性。该规则
避免把 socket、receiver、worker、relay 名称和内部进程包装成不可读天体。

下列对象不得成为行星：

- 原子事实或合并后的句子；
- episode/arc 标题和摘要；
- 天气值、状态值、布尔值、日期等孤立标量；
- source/session/turn/job/relay 等内部标识；
- 未通过命名对象校验的 `derived-*`、`agent-memory-*`、时间戳式标签；
- Vault 条目明文或可直接利用的敏感路径。

## 5. 星系：关系社区而非硬分类

### 5.1 形成依据

星系从行星之间有证据支持的关系图中产生，例如：

- 同属一个项目；
- 人物与组织、地点或经历之间的明确关系；
- 服务依赖、设备归属、部署或使用关系；
- 多个情节中反复共同出现并有事实支撑；
- 用户手动创建、命名和固定的社区。

主体恒星与大量行星天然都有联系。为避免用户恒星把全图拉成一个社区，自动社区计算默认只使用行星—行星关系；主体—行星关系用于视角、排序和召回解释，不作为社区边界的唯一依据。

每条候选边必须把三种语义分开保存：

1. **业务/依赖关系**：例如 `uses_database`、`collects_logs_from`、`connects_mailbox`；
2. **当前传输路径**：例如 `lan_direct`、`tailscale`、`local_ipc`、`oauth2_imap_smtp`；
3. **社区归属**：只是由关系图计算或人工治理得到的投影，不反向改写前两者。

局域网直连仍然存在应用—数据库业务关系，但 `LAN` 只记录为边的传输属性；只有命名且可治理的
VLAN/子网本身成为长期记忆对象时才可提升为行星。Tailscale 也只有在已验证流量实际经过它，或存在
已配置的远程/回退路径时，才参与该边并可能承担 bridge 角色；资产清单中的共同出现不能证明路径。

### 5.2 软归属与重叠

- 一个行星可以属于零个、一个或多个星系；
- `primary_galaxy` 只是当前布局的主位置，不是实体事实；
- 次级归属通过跨星系连线、边缘光晕或入口提示表达；
- 跨多个社区的桥接行星保持唯一，不在各星系复制；
- 进入任一星系查看的都是同一个 canonical entity 和同一治理记录。

`core`、`bridge`、`satellite` 是**特定星系内的投影角色**，不是实体的全局类型。同一行星可以在
数据库星系中是 satellite，在观测星系中成为日志来源，或在另一个星系中成为 bridge。

### 5.3 自动形成的最低条件

自动星系至少需要：

1. 三个可读的唯一行星；
2. 至少两条独立、可追溯的行星—行星关系链；
3. 至少一个非主体关系作为社区内聚依据；
4. 不依赖被隔离、清除或仅有低价值噪声的事实；
5. 社区名称可由锚点实体或关系主题解释。

阈值最终允许按真实数据校准，但不得为了填满主页降低到“一个实体就是一个星系”。用户手动创建的星系不受自动阈值限制，但必须标记为人工边界。

### 5.4 命名与稳定性

- 优先使用项目、地点、组织、人物关系圈等高解释度锚点命名；
- 模型只能提出候选名称和摘要，不能把名称建议写成事实；
- 人工名称、成员固定和排除规则优先于自动结果；
- 自动整理只有在关系变化超过稳定阈值并连续通过维护周期后才调整主归属；
- 每次自动移动记录旧归属、新归属、权重、输入关系和算法版本；
- 用户可撤销移动、固定行星位置或禁止某一成员再次自动加入。

### 5.5 初始语义示例

系统不预设必须存在固定六类星系，但可根据真实关系形成例如：

- Agent Memory 项目；
- 家庭基础设施与设备；
- 工作人物与组织；
- 某次旅行及相关地点；
- 兴趣、内容与创作；
- 长期生活环境。

类别只是可读名称，不是封闭本体；没有足够证据时宁可保持深空状态。

## 6. 星座、星流与事实动画

### 6.1 情节星座

情节不是星系，也不是新星体。用户选择 episode 时：

1. 高亮参与情节的恒星/行星；
2. 沿有事实支撑的关系线播放光点；
3. 在星系旁显示脱敏情节摘要、时间范围和证据数量；
4. 允许展开查看事实、来源和更正历史；
5. 关闭情节后恢复原社区布局，不改变实体归属。

只有一个行星参与的情节使用行星光晕和时间标记，不伪造第二个实体来构成图形。

### 6.2 长期脉络星流

arc 以跨时间的星座序列或轨迹表达，展示实体关系如何演变。轨迹的每一段必须能回到 episode/fact/evidence；系统不得仅为视觉连贯补写因果。

### 6.3 事实关联动画

新事实或被选中的事实可以让发光点沿其涉及实体的关系线移动。动画只表达“这些对象由该事实关联”，不表达方向、因果或置信度，除非底层关系明确包含这些语义。

## 7. 观察镜片与视觉编码

长期、阶段、当前不再是导航目的地，而是可组合的观察镜片：

| 镜片 | 默认视觉 | 数据语义 |
| --- | --- | --- |
| 长期 | 稳定、低频光晕 | 稳定或缓慢变化的有效事实 |
| 阶段 | 外层时间环/活动带 | 项目、旅行、迁移、排障等阶段事实 |
| 当前 | 脉冲和高亮 | 未过期 current item 或当前事实 |
| 情节 | 临时星座与移动光点 | 被选择或命中的 episode |
| 脉络 | 可播放星流 | 被选择或命中的 arc |
| 休眠 | 降低饱和度 | dormant 内容仍可明确检索 |
| 忘记 | 默认隐藏、显式搜索后微弱显示 | forgotten 内容不主动召回 |
| 保护 | 锁形/护盾标记 | 只存在脱敏 Vault 引用 |

镜片只改变查询范围、亮度、光晕、标签和边的可见性，不改变实体身份、星系成员关系或事实分类。

## 8. 主宇宙与子星系交互

- 首页展示主体恒星群、星系边界、深空行星和桥接关系；点击星系标签或区域直接进入；
- 连续放大星系进入子星系，子星系缩小到阈值返回主宇宙；
- 子星系默认只显示成员行星和内部/桥接关系，不重复绘制主宇宙的完整恒星群；
- 子星系标题区以非节点徽标显示“相关主体/来源 profile”，用户可按需打开主体关系光线；
- 行星默认仅在原位约 5px 范围缓慢漂移；鼠标悬停、详情打开或用户关闭动态时立即静止；
- 动静开关是用户持久偏好，并遵守操作系统的减少动态效果设置；
- 星系、行星、星座和事实注释必须拥有不同的可访问名称，Canvas 不得成为唯一操作入口。

## 9. 投影数据与可审计性

目标数据模型需要增加下列逻辑对象，具体表名可在实现设计中调整：

| 对象 | 最小字段/关系 | 约束 |
| --- | --- | --- |
| Subject | `id`, `kind=user|profile_persona`, `entity_id`, `stable_key`, `display_name`, `display_name_origin`, `status` | 指向 canonical entity；稳定键唯一；人工名称不被自动同步覆盖 |
| Subject source mapping | `subject_id`, `source_profile`, `source_instance_pattern` | 多实例可映射同一人格；禁止实例自动成星 |
| Entity alias | `entity_id`, `alias`, `source`, `confidence`, `review_state` | 先规范化再建边；合并可撤销；同实体别名不得形成自环 |
| Relation projection | `source_entity_id`, `target_entity_id`, `relation_type`, `transport`, `evidence_refs`, `state` | 业务语义与传输路径分离；共同出现不能直接升级为稳定边 |
| Galaxy projection | `id`, `name`, `anchor_entity_id`, `origin=auto|manual`, `algorithm_version`, `state` | 可重建；人工名称可固定 |
| Galaxy membership | `galaxy_id`, `entity_id`, `weight`, `role`, `evidence_refs`, `is_primary`, `manual_state` | 多对多；必须有依据或人工审计 |
| Layout preference | `target_id`, `scope`, `position`, `motion_enabled`, `pinned_by_user` | 只影响显示，不成为事实 |

关键不变量：

- `entities` 继续是唯一实体事实源；行星只是实体的 UI 投影；
- Subject 是实体的“主体角色”，不是第二份实体；
- `galaxy_id` 不写入事实或实体作为唯一硬归属；
- 自动星系、成员权重、布局和摘要均可从当前有效关系重建；
- 所有人工固定、排除、改名、合并、拆分和撤销操作进入审计；
- 生命周期更正或清除发生后，受影响社区与星座必须重建。

## 10. 用户治理

用户必须能够：

- 为用户和 profile 人格设置恒星显示名、颜色和显示状态；系统不得把 profile 类型或来源 key 自动拼入名称；
- 把 source profile/实例映射到已有恒星，或撤销错误映射；
- 创建、重命名、固定、合并、拆分、隐藏星系；
- 把行星固定到星系、标记为次级成员或排除；
- 查看自动归属依据、关系权重和最近变化；
- 撤销一次自动移动或恢复自动布局；
- 从行星详情继续执行实体合并/拆分、事实更正、脱离、忘记和清除；
- 从保护标记进入 Vault，但普通星图永不加载敏感明文。

主体映射、星系治理和布局治理不修改原始证据。人工归属表达用户的信息架构偏好，也不得被自动流程改写成现实事实。

## 11. 召回与星图的一致性

- 星系成员权重可以作为实体关系召回的解释信号，但不能覆盖明确词法匹配、证据可信度、权限和生命周期过滤；
- 星系名称命中时可扩展相关实体候选，返回时必须说明命中星系和底层关系；
- forgotten/isolated/purged 的规则先于星系扩展；星系不能让不可召回内容重新进入普通上下文；
- profile 恒星只提供来源/视角过滤，不阻止跨 profile 共享记忆；
- 星图与召回必须使用相同的 canonical entity、关系状态和 namespace 边界。

## 12. 分阶段实施计划

### 阶段 A：投影契约与身份整理

实施状态：**已完成（rc.5）**。

- 建立 Subject 与 profile/source 映射；
- 将旧 User/Hermes 固定双星迁移为主体恒星群投影；
- 清理 UUID、时间戳和内部来源标签成为实体的路径；
- 保持现有事实分类和 API 向后兼容。

阶段 A 的兼容边界是保留事实、证据、情节、脉络、Vault 与治理数据契约；历史写死的 `core:user`/`core:hermes` 只是错误的展示契约，迁移后由 `node_kind=subject` 替代，不继续作为兼容别名。`source_instance` 只出现在受控映射清单中，不产生恒星或行星。

### 阶段 B：行星渲染与观察镜片

实施状态：**已完成（rc.6）**。本阶段使用只读 `planetary-v2` 响应投影，不新增数据库表，也不写入实体归属。

- 普通 entity 从“实体恒星”改为行星；
- 移除长期/阶段/当前/情节/脉络/Vault 固定星系导航；
- 将其改为可组合镜片、星座/星流和保护标记；
- 保留搜索、详情、治理、动态开关与缩放联动。

## 13. 验收标准

1. 用户和每个配置的人格 profile 各显示为一个可读恒星；用户可使用自定义显示名，profile 恒星视觉标题只有人格名称，类型/来源标识只出现在详情、tooltip 或无障碍语义中；同 profile 多实例不会新增恒星。
2. 其他人物、项目、设备、服务、地点等只显示为行星；事实、句子和内部 ID 不显示为天体。
3. 同一行星跨多个星系仍保持一个 canonical entity 和同一治理历史。
4. 自动流程不会创建只有一个行星的星系；证据不足实体进入深空。
5. 长期、阶段、当前切换只改变镜片和统计，不改变行星归属。
6. 选择情节只生成临时星座和事实动画，关闭后不改变社区布局。
7. 星系自动形成、命名、移动和成员权重均可解释、可审计、可撤销。
8. 用户固定的主体映射、星系名称、成员和位置不会被后台整理静默覆盖。
9. 更正、脱离、忘记和清除能重建受影响的星系/星座，不残留不可召回关联。
10. 主宇宙与子星系缩放联动、约 5px 漂移、悬停静止和动静开关均通过浏览器验收。
11. Vault 明文、受保护个人资料和模型密钥不进入任何星图图元素、社区摘要或布局日志。
12. 所有页面操作具有 Canvas 之外的键盘/无障碍入口。
