# V1.0 Hermes 历史对话导入设计

## 1. 目标与边界

V1 使用真实 Hermes 历史对话作为记忆质量调优和用户验收的主要样本，避免由开发者编写的句子
主导实体、关系、阶段事实和召回效果。导入不会替代自动化测试：幂等、权限、脱敏、worker 停机、
备份恢复仍使用可重复且不含隐私的固定样本。

首期只接受 Hermes 官方 `sessions export` 生成的完整 JSONL：每行是一份包含 `messages` 的完整
session。以下内容拒绝或不导入：

- Markdown、QMD、HTML、trace；
- `--only user-prompts` 的提示词摘录；
- system prompt；
- 导入前已经由 Agent Memory 写入 Hermes 的 `agent_memory_*` 内部工具调用；
- 没有对应 user message 的孤立 assistant/tool 消息。

普通工具调用及结果属于证据，按关联事件保留；不会把工具参数或整段句子直接当作星体。

## 2. 数据流与隔离

```text
Hermes state.db（只读）
  -> hermes sessions export --format jsonl --redact
  -> 本地预览（统计、敏感扫描、SHA-256，不输出正文）
  -> hermes:import-staging（7790，按需启动）
  -> 分类/投影/召回/星图人工验收
  -> 对同一 SHA-256 显式批准
  -> hermes:user-primary（7788）
```

原始导出文件不被修改；每个 user message 开始一个 turn，后续 assistant/tool 消息归入该 turn。
幂等键由 namespace、profile、Hermes session ID 和消息/回合 ID 生成，因此同一文件重复执行只会
得到 duplicate，不会重复写入。文件 SHA-256 用于确认“预览的文件就是应用的文件”，不参与回合
身份生成，所以同一 session 在后续增量导出中仍能正确去重。

暂存服务通过 Compose `import` profile 按需启动：

- namespace 默认 `hermes:import-staging`；
- API 只监听 `127.0.0.1:7790`；
- core worker 固定处理暂存 namespace，模型默认关闭；
- 与正式库共用数据库，但所有读取、任务和投影按 namespace 隔离；
- 默认 `docker compose up` 不会启动暂存服务。

暂存不是备份，也不自动晋升。V1 的“晋升”是使用同一份已审查导出重新导入主 namespace，保留
独立证据链。暂存验证结束后可停止服务；在实现带审计的 namespace 清理前，不直接删除暂存数据。

## 3. 安全约束

1. Hermes 导出必须优先使用 `--redact`。脱敏只针对已知凭据模式，姓名、地址、项目内容等隐私
   仍可能存在，因此导出目录必须位于本机、权限受控且被 Git 忽略。
2. 导入 CLI 只允许连接 `http://127.0.0.1`、`localhost` 或 `::1`，禁止把历史正文发送给远程 API。
3. apply 必须提供与 preview 完全一致的 `--confirm-sha256`。
4. 二次扫描仍发现潜在敏感值时，必须先人工检查；确认本地风险后才可使用
`--accept-local-redaction`。该参数不表示内容可以发送给外部模型。
5. 主 namespace 还必须显式指定 `--namespace hermes:user-primary --allow-primary`，默认永远写暂存。
6. service token 只从环境变量读取，命令行和 manifest 不记录 token。
7. manifest 只保存统计与校验值，权限为 `0600`，不保存正文和工具参数。
8. 导入阶段关闭模型。后续若对历史证据启用外部模型整理，仍必须经过 Agent Memory 脱敏边界。
   规则脱敏后的正文仍可能包含个人或组织语境；只有用户对具体供应商、数据范围和风险进行知情
   授权后才可发送到外部模型。本地模型不受该外发限制，但仍必须走相同的逐字证据验证。

## 4. 操作流程

### 4.1 导出与预览

先创建只在本机保存的目录，并按 profile 分别导出。建议第一轮选最近 30 天、至少两条消息的
session；不要一开始导入全部历史。

```bash
mkdir -p data/imports/raw
chmod 700 data/imports data/imports/raw
hermes sessions export data/imports/raw/personal-30d.jsonl \
  --format jsonl --redact --newer-than 30d --min-messages 2
chmod 600 data/imports/raw/personal-30d.jsonl

uv run agent-memory-import-hermes data/imports/raw/personal-30d.jsonl \
  --profile personal
```

预览必须检查：session/turn/event 数量合理，时间范围正确，跳过数量可解释，敏感命中为 0 或已经
人工检查。不得把预览输出中的 SHA-256 写死在脚本或提交到公开问题单。

### 4.2 暂存导入

```bash
docker compose --env-file .env --profile import up -d import-api import-worker
curl --fail http://127.0.0.1:7790/health/ready

set -a
source .env
set +a
uv run agent-memory-import-hermes data/imports/raw/personal-30d.jsonl \
  --profile personal --apply --confirm-sha256 '<preview-sha256>'
```

若预览仍有敏感命中，先在本机检查导出文件并重新导出；确认为误报或已脱敏占位符后，才附加
`--accept-local-redaction`。每次应用（包括幂等重放）都会写入不可变的
`data/imports/manifests/<sha256>/<timestamp>-<id>.json`，不会覆盖第一次插入的审计结果。

### 4.3 暂存验收

打开 `http://127.0.0.1:7790/`，至少检查：

- 对话原始证据能够从记忆追溯到 session/turn，而不是多条事实共用一条伪造证据；
- 星图只显示实体，句子和派生事实不成为星体；
- 长期、阶段、当前和低价值分类符合真实语境；
- 普通工具结果可追溯，`agent_memory_*` 内部调用没有形成自反馈记忆；
- 密钥、密码等没有进入普通事实、召回或星图；
- 相同命令重跑后 `inserted_turns=0` 且 `duplicate_turns` 等于计划回合数；
- 跨 profile 的同一用户身份能够合并，来源 profile 仍可追溯。

检查完成后停止暂存服务：

```bash
docker compose --env-file .env --profile import stop import-api import-worker
```

### 4.4 主库导入

只有暂存验收通过且已完成数据库/Vault 备份时执行：

```bash
backup_dir="$(bash scripts/backup.sh .env)"
set -a; source .env; set +a
uv run agent-memory-import-hermes data/imports/raw/personal-30d.jsonl \
  --namespace hermes:user-primary --profile personal \
  --api-url http://127.0.0.1:7788 --apply \
  --confirm-sha256 '<preview-sha256>' --allow-primary
```

主库导入后等待 worker 队列归零，再进行召回、星图和证据追溯抽检。任何分类问题应通过更正和
派生重建处理，不修改原始导出或只读证据。

## 5. 验收门槛与后续改进

首轮建议抽取 20 至 50 个真实 session，覆盖项目开发、排障、旅行/阶段经历、长期偏好、普通问答
和包含敏感值的已脱敏场景。达到以下门槛再扩大范围：

- 导入幂等率 100%；
- 原始 turn 可追溯率 100%；
- 已知凭据进入普通事实/召回为 0；
- 内部 memory 工具形成自反馈事实为 0；
- 用户抽检的关键事实分类与实体归属可接受，错误均可在星图更正并重建。

社区实践上不建议一次性全量导入：真实历史中有大量过期配置、重复排障和上下文依赖表达，直接
全量固化会扩大错误实体合并与陈旧事实的影响。V1 应采用“小批量暂存—报告—人工抽检—扩大批次”
的渐进方式；后续再增加按 workspace/session 选择、批次回滚、暂存 namespace 清理和导入质量报告。

## 6. 2026-07-15 真实数据基线

使用 `qishuo` profile 最近 7 天、至少 2 条消息的 Hermes 官方脱敏导出完成首轮隔离验证，未写入
`hermes:user-primary`：

| 项目 | 结果 |
| --- | ---: |
| session / turn / event | 84 / 89 / 2384 |
| user / assistant / tool call / tool result | 89 / 89 / 1103 / 1103 |
| 二次敏感命中 | 97（凭据样式 89、身份证格式 8） |
| 幂等重放 | 0 inserted / 89 duplicate |
| worker 最终任务 | 3673 done / 0 pending / 0 failed |
| 确定性事实 / 实体 / 检索文档 | 32 / 1 / 32 |

真实批次发现并修复了两个合成测试未充分暴露的问题：

1. manifest 重放会覆盖第一次插入审计，已改为同一 SHA-256 下每次 apply 生成不可变文件；
2. JSON 工具参数中的敏感字段和内容命中未完整进入持久化前脱敏/审计，已改为结构化递归脱敏。
   修复后预览 97 项与数据库 `evidence.redaction_findings` 97 项完全一致。
3. 模型关闭的 worker 仍因顶层导入 LiteLLM 尝试联网获取价格表，已改为仅在真实模型任务构造
   adapter 时按需导入；重启日志只包含 `worker started`，模型关闭路径不再产生模型网络请求。

该基线也证明当前确定性 Scribe 不能作为真实历史的最终抽取器：84 个 session 只形成 1 个实体，
而 32 条事实仍以整条来源消息为单位，不满足 V1 的原子事实和实体星图质量目标。因此本批次只作为
安全与导入基线保留，不批准进入主 namespace。下一开发优先级是“证据逐字约束的多原子事实与实体
抽取”，并以该批次作为离线评测集；在其完成前继续增加星图动画或扩大历史导入都不会改善记忆质量。

## 7. 2026-07-16 原子抽取与隔离审计

已实现按 Hermes turn 的原子抽取任务 `extract_atomic_turn`。每个候选必须指定 evidence index，事实文本
必须是对应脱敏证据中的连续逐字片段，实体名称必须是该事实片段中的连续逐字子串；事实、实体提及、
事件 ID 和字符跨度均持久化。模型返回正文不落库，审计只保存模型标识、计数、版本和结果。

真实基线的只读选择规划结果：89 个 turn 均有可用用户消息，因此调用上限为 89，而不是按 1103 条
工具结果逐项调用。初版启发式曾选中 31 条工具结果，其中 29 条来自 `vision_analyze`，另两条为模型
聊天与 OpenAPI 规范；据此改为显式工具允许列表。默认仅 `terminal`、`exec`、`execute_code` 可作为
模型观察证据，本批次最终选择为 89 条用户消息、0 条工具结果、每 turn 1 条证据。

同时修复了 Worker 领取任务未限制 namespace 的隔离缺陷。核心、模型、导入和自动化 Worker 现在只
领取自身 namespace 的任务；自动化回归会启动专属 API 与专属 Worker。运行时用 staging/canary 两个
非主 namespace 验证，staging 任务完成而 canary 保持 pending；最终完整 Release Gate 再次通过。

原子链路的数据库自检已验证 2 条事实、2 条实体提及和 2 个检索文档。真实数据外部评测必须由用户对
具体 namespace、固定样本计划和供应商单独知情授权；未授权时模型 Worker 保持关闭。

外部供应商兼容性已使用完全虚构的 Aurora/PostgreSQL 样本验证：OpenCode Go
`openai/deepseek-v4-flash` 通过 custom base URL 返回 3 条候选，逐字验证接受 3 条、拒绝 0 条；随后
使用正式 worker 提示词完成模型到 PostgreSQL 的端到端冒烟测试，落库 2 条原子事实和 2 条实体提及。
测试只写入 `hermes:automated-tests:model-smoke`，模型 Worker 对主空间和真实 staging 仍保持关闭。

真实评测采用固定的小批量计划，而不是临时挑选正文。`agent-memory-plan-evaluation` 默认从 staging
的含用户消息 turn 中，以固定 seed 对规则信号分层并轮询抽取 24 个 turn；默认排除任何曾触发脱敏规则
的 turn，只有明确批准的本地评测才可使用 `--include-redacted-turns` 纳入。输出只包含 turn ID、profile、
时间、分层、脱敏命中计数和确认 SHA，不含对话正文，不调用模型，也不发送外部数据。评测执行必须绑定
同一 SHA；若计划、数据或 seed 变化则必须重新确认。首轮只允许用户消息作为模型 evidence，不包含工具
结果、Vault、附件或主 namespace。结构合法率、逐字支撑率、分类准确度和实体准确度仍需人工抽检，
不得用规则分层本身充当模型质量真值。

隔离 model worker 的评测模式还必须同时配置计划 SHA 和精确 turn UUID allowlist；领取 SQL 只会领取
allowlist 中的 `extract_atomic_turn`，backfill 也只会为这些 turn 建任务。评测模式禁止主 namespace。
非自动化 namespace 使用公网模型端点时，还必须显式设置 `AGENT_MEMORY_MODEL_ALLOW_EXTERNAL_DATA=true`；
默认值为 false，本地回环、私网 IP、`.local` 和 Docker 单标签服务名不需要放宽该边界。

因任一事件存在脱敏命中而排除，剩余 69 个可选；固定抽取 24 个，分布为 candidate 6、current 2、
evidence-only 6、low-value 5、long-term 5，选中样本脱敏命中为 0。计划确认 SHA 为
`7a30aafed367b2186c834f69dcd81d0dd1f5dac0c7dab4ba293c58e3752f9d48`。运行结果明确记录
`contains_memory_text=false`、`model_called=false`、`external_data_sent=false`。2026-07-16 用户随后明确
授权该 SHA 对应的 24 条 staging 对话使用 OpenCode Go DeepSeek V4 Flash 评测。

为兼容既有整句事实，召回层增加同证据重叠抑制：当某条 `model-verbatim` 原子事实与

episode/arc；因此既保留审计链，又避免 Hermes 被同一内容重复注入。
