# Agent Memory for Hermes — V1.0 技术选型方案

## 1. 结论摘要

V1 推荐采用单一、可恢复的关系型数据核心，而不是拆分为图数据库、向量数据库、消息队列和多个模型网关。

```text
Hermes Memory Provider（Python 插件）
            │ localhost authenticated API
            ▼
Agent Memory API（FastAPI） ── 星图 Web UI（React + Cytoscape.js）
            │                         │
            └──── PostgreSQL 18.x + pgvector ────┘
                         ▲
                         │ durable jobs / audit / vault / graph relations
            独立 Worker（Python）
            ├─ 证据处理、索引、整理、状态计算
            └─ LiteLLM SDK → 外部 API / 本地模型 / 自定义 Base URL
```

推荐 Docker Compose 服务数量为三项：`postgres`、`api`、`worker`。星图前端编译为静态资源并由 API 服务提供；本地模型服务不随本项目强制部署，而是作为可配置 endpoint 使用。

## 2. 选型目标与约束

- 单用户、单 Hermes、多 profile 共享统一记忆命名空间。
- 长期本地运行：证据、审计、凭据元数据、派生记忆必须具备强一致性和可恢复性。
- 词法、语义、实体关系三路召回必须在同一权限、生命周期和事务边界内融合。
- Docker 部署于 Apple Silicon 本机；当前宿主机为 Apple M4、16 GB 内存。
- 外部 API、局域网模型及本地模型必须可通过配置切换；敏感明文不得发送至任一模型。
- 不为 V1 引入多用户、高吞吐分布式集群所需的额外组件。

## 3. 最终推荐与备选对照

| 层 | 推荐 | 不选/延后 | 理由 |
| --- | --- | --- | --- |
| Hermes 接入 | Python `MemoryProvider` 插件 | 单纯 shell Hook、修改 Hermes 核心 | 原生拥有预取、回合同步、工具路由、会话切换和 Provider 生命周期 |
| 核心数据库 | PostgreSQL 18.x | SQLite 为长期主库 | 同时承载证据、审计、关系、任务和向量，便于事务、备份与并发 worker |
| 向量检索 | pgvector | 独立向量数据库 | 记忆规模与单用户场景无需独立集群；过滤、来源和生命周期可在同一查询/事务中处理 |
| 关系图 | PostgreSQL 实体/边/证据表 | Neo4j 等独立图数据库 | V1 的图主要用于实体关联和星图，不值得引入双写、一致性和备份复杂度 |
| 词法检索 | PostgreSQL FTS + `pg_trgm`，经可替换分词层标准化 | 仅向量检索 | 命令、服务名、项目名、日期和中文关键词需要可解释的精确/模糊匹配 |
| API | FastAPI + Pydantic | GraphQL、BaaS | Python 与 Hermes Provider/worker 同语言；契约和 OpenAPI 适合本地服务边界 |
| 异步整理 | 独立 Python worker + PostgreSQL 持久任务表 | FastAPI 进程内后台任务、先引入 Redis/Celery | 整理任务需要重试、幂等、审计和重启恢复；V1 以最少组件获得持久队列能力 |
| 模型路由 | LiteLLM Python SDK + 本项目 Provider 抽象 | V1 即部署 LiteLLM Proxy | SDK 已覆盖外部/本地/custom base URL；单用户先避免额外网关与管理面 |
| 敏感信息库 | PostgreSQL 独立 schema + 应用层信封加密 | 普通表明文、把凭据写入模型配置 | 密钥和个人资料必须从普通记忆、索引、日志和模型请求中隔离 |
| 星图前端 | React + TypeScript + Cytoscape.js | 服务端渲染大图、桌面专用技术 | 浏览器端图布局与交互成熟，且可逐步实现上游的星图体验 |

## 4. Hermes 集成架构

### 4.1 Provider 职责

`agent_memory` 作为唯一外部 Memory Provider，保留 Hermes 内建 `MEMORY.md` / `USER.md`，并额外承担：

- `initialize`：将 Hermes profile 映射为来源 metadata，而非记忆隔离边界。
- `on_turn_start` / `prefetch`：向本地 API 请求受预算限制的轻量召回。
- `sync_turn`：异步提交用户消息、助手回复和工具结果作为证据候选。
- `on_session_switch` / `on_session_end` / `on_pre_compress`：写入会话边界并触发整理。
- Memory Provider tools：实现召回、浏览、来源追溯、当前状态更新和更正。

### 4.2 跨 profile 命名空间

Hermes profile 原生隔离；Agent Memory 必须显式配置一个稳定的 `shared_namespace`，例如 `hermes:<用户标识>`。

- `shared_namespace`：V1 的真实长期记忆边界。
- `source_profile`：证据和操作来源，用于筛选与审计。
- `scope`：全局、项目、会话/阶段、私密、禁止召回。
- Hermes profile 不得直接决定 Agent Memory 的物理存储路径。

## 5. 数据层

### 5.1 PostgreSQL 18.x + pgvector

选择 PostgreSQL 作为唯一事实源，使用事务维护证据、派生记忆、关系、任务、审计和敏感资源元数据的一致性。

PostgreSQL 内建全文检索可提供词法候选；pgvector 支持向量相似度索引，并明确支持与 PostgreSQL 全文检索进行混合召回与 RRF 融合。V1 先使用精确向量检索与小规模 HNSW 对照基准，再根据真实数据规模决定默认索引策略。

### 5.2 初始逻辑 schema

| schema / 表族 | 关键内容 |
| --- | --- |
| `evidence.events` | 只读来源事件、会话/turn/tool 关联、脱敏正文、完整性校验、保留策略 |
| `memory.facts` | 原子事实、认知状态、置信度、有效期、替代链 |
| `memory.entities` / `memory.relations` | 实体、别名、关系、关系强度、自动合并依据 |
| `memory.episodes` / `memory.arcs` | 派生叙事、支撑事实、版本与重建状态 |
| `retrieval.documents` | 可检索投影、词法索引字段、向量、索引版本、生命周期状态 |
| `state.*` | 当前状态、会话连续性、连续互动状态、状态事件与快照 |
| `vault.entries` / `vault.grants` | 加密密文、脱敏摘要、授权、轮换与使用审计 |
| `ops.jobs` / `ops.job_attempts` | 证据处理、整理、重建、索引和报告的持久任务 |
| `audit.*` | 用户操作、自动整理、召回、敏感访问和配置变更 |

原始证据不被派生层覆盖；派生记录一律持有来源 ID 与版本信息。永久清除在单个受控工作流中删除或加密销毁对应证据、投影、关系、索引和审计中的允许删除部分。

### 5.3 词法、语义和关系召回

- **词法**：PostgreSQL FTS 用于英文/结构化文本；`pg_trgm` 支持服务名、命令、项目名与中文短语的模糊/子串候选。分词层必须可替换，避免将中文分词器绑定进数据库核心。
- **语义**：将可检索投影的 embedding 保存至 `pgvector`；按 namespace、作用域、权限、状态和时间先过滤，再执行近邻检索与重排。
- **关系**：从实体、关系边、项目归属、会话和时间线扩展候选。
- **融合**：在应用层执行可配置 RRF 和业务重排，返回各通道分数、来源和最终解释。

### 5.4 不采用图数据库的退出条件

以下任一条件持续出现时，才重新评估独立图数据库：关系遍历成为绝大多数查询瓶颈；单次需进行多跳复杂图算法；或图数据规模、协作用户数和跨域权限超过 PostgreSQL 关系表可合理维护的范围。

## 6. 服务与后台任务

### 6.1 API 服务

采用 FastAPI，负责：

- Hermes Provider 的本地 API；
- 星图、整理报告、状态面板和受保护资源 API；
- 管理操作、作用域检查、审计与健康检查；
- OpenAPI 契约，供 Provider 与前端在不共享进程的情况下演进。

Provider 与 API 的初始通信方式为只监听 `127.0.0.1` 的 HTTP，使用安装时生成的本地服务令牌认证。未来 Hermes 也容器化后，可升级为 Docker 内部网络或 Unix socket，不改变 API 契约。

### 6.2 持久 worker

采用与 API 分离的 Python worker，从 `ops.jobs` 拉取任务。任务必须具备：

- 幂等键、输入版本、状态机、有限重试和退避；
- 崩溃恢复、死信/人工处理状态；
- 写入审计和整理报告；
- 深度整理与模型调用的速率、并发和成本控制。

不得用 FastAPI `BackgroundTasks` 承担证据整理、LLM 调用、重建或定期维护等必须跨重启存活的工作。

## 7. 模型与 embedding

### 7.1 路由方案

采用 LiteLLM Python SDK 封装模型调用；每个任务引用一个 `model_profile`，可设置模型、提供商、`api_base`、超时、重试、最大成本/Token 和敏感数据策略。

| 任务 | 默认策略 |
| --- | --- |
| 事实候选提取 | 低成本结构化输出模型；失败时只保留证据并延后重试 |
| 情节/长期脉络整理 | 更强模型、低频批处理、严格证据提示 |
| 对话状态增量 | 轻量模型或规则；失败时不产生自主行为 |
| Embedding | 独立 embedding profile；可指向本地或外部 endpoint |
| 普通检索 | 不依赖 LLM；模型不可用仍可进行词法和关系召回 |

LiteLLM SDK 可统一外部 API、Ollama 和自定义 `api_base`；V1 使用 SDK 而不是 Proxy 服务。只有需要跨多个应用集中治理、配额、虚拟密钥或统一模型审计时，才将 LiteLLM Proxy 作为可选第四个 Compose 服务引入。

### 7.2 本机模型边界

Apple M4、16 GB 内存适合进行轻量本地 embedding、规则处理与小模型验证；不应将大参数抽取模型作为 V1 可用性前提。默认策略是：

- 本地 embedding 优先作为基础能力；
- 事实提取与深度整理可选外部 API 或用户已有局域网模型；
- 外部调用前执行敏感值剔除和最小化上下文；
- 通过基准测试确定最终 embedding 模型、抽取模型和并发上限，不在本文硬编码具体模型名称。

## 8. 敏感信息库

### 8.1 加密与密钥边界

- `vault` schema 仅保存加密密文、脱敏摘要、用途、关联对象和授权策略。
- 每条敏感条目使用独立数据加密密钥；数据密钥由不写入数据库的根密钥包装。
- 根密钥通过 Docker secret 或受限挂载文件提供，禁止放入 Compose 文件、普通 `.env`、日志或普通备份明文。
- 密钥版本、轮换和失效均可审计。

### 8.2 使用代理

Hermes 永远不通过普通召回取得敏感明文。一次授权应生成“对某目标执行某操作”的短期能力；API/worker 代理使用凭据并仅向 Hermes 返回脱敏结果。直接明文披露是用户手动操作，默认关闭且必须审计。

## 9. 星图前端

采用 React + TypeScript + Cytoscape.js。

- API 返回按需加载的节点、边、时间线、状态、来源和治理操作，不把整个记忆库一次性发送到浏览器。
- Cytoscape.js 负责交互图元素、筛选、局部子图布局、关系高亮和节点详情联动。
- 星图将“受保护资源”显示为脱敏关联，详情跳转到独立 Vault 页面；不会把敏感值变成图元素属性。
- V1 视觉上复用 Memory Constellations 的“核心—星系—星座—桥接”信息架构，而不是复制其实现。

## 10. Docker、备份与可观测性

### 10.1 Compose 拓扑

| 服务 | 职责 | 对外暴露 |
| --- | --- | --- |
| `postgres` | 主数据库、pgvector、全文/关系/任务/审计/Vault 元数据 | 仅 Docker 内网 |
| `api` | Provider API、星图 API、静态前端、健康检查 | `127.0.0.1` 本地端口 |
| `worker` | 整理、索引、状态、报告、模型任务 | 无 |
| `litellm-proxy` | 可选，非 V1 默认 | 仅 Docker 内网 |

镜像必须支持 `arm64`，版本与镜像 digest 固定；数据库数据、Vault 根密钥挂载、配置和备份路径必须位于项目目录或用户明确的持久卷中。

### 10.2 备份与恢复

- 每日逻辑备份使用 `pg_dump` 生成可迁移归档；周期性执行恢复演练和完整性验证。
- 不把数据库卷的文件复制当作唯一备份方式。
- 备份必须包含 PostgreSQL 数据、项目配置、Docker Compose 文件、Vault 根密钥恢复材料的独立安全副本，以及模型/Provider 配置的脱敏版本。
- 恢复后需验证证据数、派生版本、任务状态、索引重建能力和 Vault 条目可读性。

### 10.3 运行信号

- JSON 结构化日志：请求、作业、模型、审计和错误均含 correlation ID。
- 健康检查：API、worker 心跳、数据库连接、迁移版本、待处理/失败任务数。
- 指标：召回延迟与命中通道、作业耗时/重试、模型错误与成本、证据积压、Vault 拒绝访问数。

## 11. 技术验证计划（第二个 Spike）

在正式实现前，创建仅含 `postgres + api + worker` 的第二个 Spike，验证：

1. 迁移能够创建证据、事实、实体、关系、检索和作业 schema。
2. 同一 `shared_namespace` 下两个 profile 的证据可跨 profile 召回，并保留来源 profile。
3. 词法、语义、关系三路候选可融合为带证据解释的召回结果。
4. 重复提交同一 Hermes turn 不产生重复证据或稳定事实。
5. worker 重启后能恢复未完成任务；更正与永久清除能传播至投影和索引。
6. 虚构敏感 Token 经由事件写入后，只存在脱敏证据，且永不进入检索投影。
7. `pg_dump` 备份可在空实例恢复并通过基本数据核验。

## 12. 决策状态

### 已建议冻结

- Hermes 外部 Memory Provider 插件。
- PostgreSQL 18.x + pgvector 作为单一数据核心。
- FastAPI API + 独立 Python worker。
- PostgreSQL 持久任务表，不引入 Redis/Celery。
- React + TypeScript + Cytoscape.js 星图。
- LiteLLM SDK（非 Proxy）作为模型适配层。
- Vault 独立 schema、信封加密、操作能力授权。

### 进入基准测试后冻结

- 具体 embedding 模型与维度。
- 本地抽取模型是否满足质量与延迟；否则使用哪类外部 API。
- HNSW / 精确检索的阈值、索引参数与召回预算。
- 默认 worker 并发、整理触发间隔和模型成本上限。

## 13. 主要风险与控制

| 风险 | 控制 |
| --- | --- |
| 中文/命令检索效果不稳定 | 保留词法、语义和关系三路；用真实 Hermes 历史构建基准集 |
| 外部模型不可用或泄露数据 | 模型可替换、敏感剔除、作业延后重试、本地词法/关系召回降级 |
| 自动整理制造错误记忆 | 证据约束、候选状态、冲突并存、用户更正与可重建派生层 |
| 任务重试导致重复写入 | 幂等键、输入版本、事务和作业审计 |
| 跨 profile 误召回 | 显式 namespace + profile 来源 + scope/权限过滤 |
| Vault 变成模型秘密通道 | 明文隔离、最小能力授权、代理执行与审计 |
