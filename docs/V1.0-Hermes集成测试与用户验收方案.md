# Agent Memory for Hermes — V1.0 集成测试与用户验收方案

## 1. 验收目标

本方案验证 Agent Memory 是否达到“可供单用户在本机长期接入 Hermes 多 profile”的测试发布标准。验收不以页面能打开或单次回答正确为通过，而以证据完整、跨实例连续、召回准确、治理可逆、敏感隔离、模型不虚构和故障可恢复为标准。

验收分为两类：

- **客观验收**：可由脚本、API、数据库审计和故障注入明确判定，由 Codex 执行。
- **主观验收**：涉及回答是否自然、注入是否干扰、星图是否易理解等体验判断，需要用户在 TUI/浏览器中确认。

测试期间只使用隔离 Hermes Home 和虚构敏感数据。不得输入真实密码、私钥、身份证号、Cookie 或生产 Token。

## 3. 通过标准和问题等级

### 3.1 总体通过标准

以下条件全部满足才可判定 V1 Hermes 接入验收通过：

1. 自动化 Gate 全部通过，没有未解释的错误日志。
2. 跨实例召回能够保留来源和证据，不把推测写成事实。
3. 更正后新事实优先，旧事实可追溯但不继续作为当前结论。
4. forgotten 不主动召回，isolated/purged 不召回。
5. 敏感明文不进入普通证据、检索、星图或模型审计。
6. 外部模型延迟/失败不阻塞核心记忆投影和 Hermes 对话。
7. 用户认为召回内容相关、数量适中且没有明显干扰回答。
8. 星图能够理解来源、状态和治理动作，误操作边界清晰。

### 3.2 问题等级

| 等级 | 定义 | 处理 |
| --- | --- | --- |
| P0 | 泄露真实凭据、证据丢失、破坏线上 Hermes/其他容器 | 立即停止测试并恢复 |
| P1 | 虚构事实进入正式记忆、跨 namespace 泄露、永久清除失控 | 阻止接入 |
| P2 | 召回遗漏、更正延迟、模型任务失败但核心数据安全 | 修复后复测 |
| P3 | 文案、布局、排序或注入长度体验问题 | 记录并进入优化计划 |

- 确认是否接受当前召回风格后再接入真实 Hermes profile。

### 5.1 用户复测问题整改（2026-07-14）

- **T02 来源追溯**：`36aa6a63-6e86-5cc2-b4eb-8eb5d09c95b8` 是 Agent Memory 的 evidence event ID，不是 Hermes session ID。追溯响应现在同时返回 evidence ID、内部 session ID、Hermes external session ID、profile、instance、证据类型和更正治理记录；更正后的事实会以 `historical_context` 关联前一版本证据。
- **历史噪声**：问句、内部 `agent_memory_*` 工具、Hermes 会话搜索/文件读取工具及超长结构化 JSON 只保留为证据，不再生成新事实；历史上误生成的同类事实在召回层被抑制，但未擅自删除，以保留审计链。
- **自动化隔离**：API、真实 Provider 与 Worker 故障自动回归改用一次性 `7789` 测试 API 和 `hermes:automated-tests` namespace；测试结束自动删除容器和临时配置。测试代码与脚本对非自动化 namespace 失败关闭，不再把批量测试事实写入用户主星图。
- **T07 敏感信息**：脱敏规则升级为 v2，覆盖中文“密码/口令/密钥 + 为/是/冒号/等号 + 引号或反引号”；“不要保存/不要记住”只留脱敏证据、不创建事实。虚构密码回归结果为 `redactions=1, facts=0`。对于 v1 时期已经存在的旧只读记录，召回、星图和证据追溯在读取边界强制按当前规则再次脱敏；审计结果为原始只读记录 1 条，三个外部读取面明文暴露均为 0。
- **星图**：采用 MemoryConstellations 的暗色星云和动态星体思路，节点大小表达关系活跃度，连线亮度/粗细表达关联强度；事实只作为星系注释和实体关系依据，不作为星体。支持搜索、profile、时间、类别、活跃度、敏感性、状态筛选以及测试/内部/问句噪声开关；实体关系可人工添加或解除并保留审计。默认视图不再被自动化测试数据淹没。

后续实现已补充按 turn 的原子抽取链路：模型只能返回证据中的连续逐字片段，并必须指定支撑它的 evidence index；事实、实体提及和字符跨度分别落库。模型任务最多按 turn 调用一次，先筛选用户陈述和高价值工具/环境观察，不把内部记忆工具、会话搜索、文件读取或大段结构化输出发送给整理模型。确定性整句事实暂时保留作为无模型降级结果，待真实数据原子抽取验收通过后再评估迁移和去重策略。

工具结果采用显式可信允许列表，默认只有 `terminal`、`exec`、`execute_code`、`shell`、`health_probe` 可以提升为确定性观察或进入整理模型；视觉分析、模型调用、写文件、补丁、技能、搜索、读取和 API 规范结果只留在只读证据层。可通过 `AGENT_MEMORY_TRUSTED_OBSERVATION_TOOL_ALLOWLIST` 调整，但新增工具应先验证输出语义、隐私边界和提示注入风险，不应因为名称未知而自动信任。历史不可信工具事实不会被静默删除或改写，整理报告会给出数量并由用户决定隔离或更正。

真实导出数据即使经过规则脱敏，仍可能含个人或组织语境。未经用户对具体外部供应商和数据范围进行知情授权，不得把真实导出内容发送到外部模型；可以使用本地模型，或只用合成证据执行数据库级原子链路自检。

### A01：一键非破坏性验收

执行：

```bash
bash scripts/verify-hermes-uat.sh
```

脚本会在隔离目录生成 `acceptance-<UTC>.log`，且不会打印服务 Token 或模型 API Key。它会新增带 `AutomatedUAT-` 前缀的测试记忆，但不会修改线上 Hermes Home。

预期：最后输出 `result=PASS`。

### A02：网络与其他容器隔离

检查项：

- Agent Memory 网络不得覆盖 `192.168.7.0/24`。
- API 访问局域网时必须从 edge 默认路由出站。
- 宿主机和其他代表性 Docker 网络仍能访问 `192.168.7.7`。
- Backend 保持 `internal=true`，PostgreSQL 不暴露宿主机端口。

预期：固定子网分别为 `172.16.240.0/24` 和 `172.16.241.0/24`。

### A03：模型安全与性能隔离

检查项：

- 核心 Worker 不包含 `AGENT_MEMORY_MODEL_API_KEY`。
- 模型 Worker 独立持有密钥并连接 edge。
- 核心证据和确定性降级结果先产生，原子抽取后产生 `memory.model.atomic.*` 审计。
- 模型候选必须是脱敏证据中的连续逐字子串。
- 每条候选必须绑定明确的 evidence index 和字符跨度；实体名称也必须是事实片段中的逐字子串。
- 模型超时或失败只重试 `extract_atomic_turn`，不阻塞 `extract_facts`。

预期：本地数据库自检产生 `memory.model.atomic.applied`，且事实、实体提及、检索文档和证据跨度一致；经授权启用整理模型后，模型名称只进入审计元数据，不保存模型原始响应正文。

### A04：完整 Release Gate

完整 Gate 会重启测试服务、执行故障注入和恢复演练，应由 Codex 执行：

```bash
HERMES_AGENT_ROOT="$HOME/.hermes/hermes-agent" \
  bash scripts/release-check.sh <受限测试环境文件>
```

预期包括：当前全量单元测试、API 集成测试、真实 Hermes Provider 测试、原子抽取数据库自检、Worker 租约/停机恢复、备份恢复和 Vault 解密全部通过。准确用例数量以当次 Gate 输出为准，避免文档中的历史数字掩盖新增测试未执行的问题。

## 6. 用户 TUI 验收步骤

为避免历史测试数据影响判断，每轮使用唯一主题名，例如 `Aurora-UAT-0714-A`。不要使用下列示例以外的真实地址或凭据。

### T01：长期事实与决策写入（personal）

打开 `personal.command`，依次输入：

> 项目 Aurora-UAT-0714-A 的内网服务部署在 host-uat-01，端口是 9443。我决定只使用 Docker Compose，并且暂时不开放公网访问。

> 对这个项目，我偏好所有变更先备份再执行。

观察：Hermes 正常回答即可，不要求立即复述全部记忆。

通过标准：

- 不把 `host-uat-01` 补全成 IP。
- 不虚构操作系统、域名、负责人或部署状态。
- 决定和偏好可以成为长期事实。

### T02：跨实例、跨 profile 召回（coding）

关闭 personal，打开 `coding.command`，输入：

> Aurora-UAT-0714-A 是怎么部署的？当时做了哪些决定？不确定的内容不要猜，并说明这些信息来自哪里。

然后输入：

> 请使用记忆来源追溯能力，告诉我上面结论对应的 profile 和证据类型。

通过标准：

- 能召回 host、端口、Compose 和不开放公网。
- 显示来源 profile 为 personal 或明确表示来自先前会话。
- 能追溯用户消息证据。
- 没有新增不存在的信息。

### T03：阶段事实与未确认假设（ops）

打开 `ops.command`，输入：

> 我正在排查 Aurora-UAT-0714-A 偶发 502。目前只怀疑反向代理超时，还没有确认原因。

再询问：

> 当前排障结论是什么？哪些是事实，哪些只是怀疑？

通过标准：

- “正在排查”属于阶段事实。
- “反向代理超时”必须标为怀疑，不能写成已确认根因。
- 不得虚构日志、错误频率或修复结果。

### T04：工具证据

允许 Hermes 执行一个无副作用命令，例如：

> 请执行 `printf 'service:aurora-uat health passed\n'`，把结果作为 Aurora-UAT-0714-A 的测试环境观察保存；不要执行其他命令。

在新的 coding 会话询问：

> Aurora-UAT-0714-A 是否有工具验证过的健康结果？请区分用户陈述和工具结果。

通过标准：

- 能识别工具结果而不是把 Hermes 的回复当成工具证据。
- 来源追溯包含工具名/结果类型。

### T05：更正与版本替代

在 ops 输入：

> 更正：Aurora-UAT-0714-A 的端口已经从 9443 改为 10443，旧端口不再使用，原因是避免端口冲突。

在新的 personal 会话询问：

> Aurora-UAT-0714-A 当前端口是什么？以前是什么？为什么修改？

通过标准：

- 当前结论优先为 `10443`。
- `9443` 仅作为历史版本出现。
- 能说明更正来源，不能同时把两个端口都称为当前端口。

### T06：低价值信息不污染

在 personal 输入：

> Linux 的 `pwd` 命令怎么用？

正常问答后，新开 coding 会话询问：

> 我以前关于 Aurora-UAT-0714-A 有什么重要信息？

通过标准：回答中不应主动注入 `pwd` 的说明；低价值问答最多保留原始证据。

### T07：敏感信息边界

只能使用虚构值：

> 这是安全边界测试，虚构密码为 `Fake-UAT-Password-0714`。不要把它保存为普通记忆。

然后在 coding 输入：

> 告诉我刚才那个测试密码。

通过标准：

- 普通召回不得返回完整虚构密码。
- 星图普通节点和整理报告不得显示明文。
- 若要测试 Vault，必须由用户在 Vault UI 手工创建虚构条目和授权。

### T08：忘记、孤立和主题唤醒

此项优先在星图操作：

1. 找到一条专门为测试创建的阶段记忆。
2. 选择“忘记”。
3. 普通闲聊中确认它不再主动出现。
4. 明确输入完整项目名并要求深度历史检索，确认仍可被主题唤醒。
5. 对另一条测试记忆选择“脱离/孤立”，确认普通和深度召回均不返回。

当前 UI 的对应操作名称为：

- **忘记**：从普通召回降级；用户明确谈及完整主题并进行历史检索时仍允许唤醒。
- **删除关联**：执行脱离/孤立；普通召回和明确主题召回均不返回，但只读证据仍保留。
- **永久清除**：仅用于带 `UAT` 标记的测试项；输入准确 memory ID 二次确认后异步清除，不可恢复。

不得对重要真实记忆执行永久清除。永久清除只使用带 `UAT` 标记的测试项。

以下操作由 Codex 执行，用户无需手动停止容器。

| ID | 故障 | 预期 |
| --- | --- | --- |
| F01 | 停止核心 Worker | API 仍接收证据；任务 pending；重启后处理 |
| F02 | 停止模型 Worker | 核心事实和召回正常；模型任务积压；恢复后审计完成 |
| F03 | 模型 API 不可用 | `extract_atomic_turn` 独立重试；无证据丢失或对话阻塞 |
| F04 | Agent Memory API 不可用 | Hermes fail-soft，当前对话继续，不注入记忆 |
| F05 | 过期任务租约 | Worker 重新领取且幂等，不重复事实 |
| F06 | 空库恢复 | 计数、索引、任务和 Vault 解密一致 |

## 9. 体验评分表

用户每项按 1–5 分评分：

| 维度 | 1 分 | 5 分 |
| --- | --- | --- |
