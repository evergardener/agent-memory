# Agent Memory for Hermes — V1.0 Vault 安全设计

## 1. 边界

Vault 是星图产品内的独立受保护子域。普通记忆、证据、检索投影、embedding、模型请求和星图图元素均不得保存 Vault 明文。普通记忆只允许保存条目 ID、用途、显示标签和用户提供的脱敏提示。

V1 允许的模型明文操作只有 `reveal_to_model`，且必须同时满足：用户已手工创建 grant、指定准确 Hermes profile、grant 未撤销、未过期且最长不超过 24 小时。默认无 grant；拒绝和允许均进入追加审计。

## 2. 加密格式

每条 Vault 记录随机生成 256-bit DEK：

1. DEK 使用 AES-256-GCM 加密条目明文，随机 96-bit nonce；AAD 绑定 entry ID 与 kind。
2. 根密钥使用 AES-256-GCM 包装 DEK，使用独立随机 96-bit nonce；AAD 绑定 entry ID 与 key version。
3. PostgreSQL 只保存 ciphertext、两个 nonce、wrapped DEK 与 key version。
4. 根密钥是 32 字节随机值的 Base64 编码，只从 `/run/secrets/vault_root_key` 读取。

复制密文到其他条目、修改 kind/ID、替换 nonce 或使用错误根密钥都会导致 GCM 认证失败，而不是产生未经验证的明文。

## 3. 根密钥与启动

```bash
bash scripts/init-local.sh
docker compose up --build -d
```

初始化脚本不会覆盖现有 `.env` 或根密钥。两者均位于 Git ignore 范围；根密钥通过 Compose secret 只挂载到 API。数据库、worker、镜像、普通 `.env` 和构建上下文均不包含根密钥。

根密钥丢失意味着 Vault 密文不可恢复。用户必须把根密钥恢复材料单独保存到安全位置，不得只依赖数据库备份，也不应与数据库归档保存在同一普通目录。

## 4. API 与 Hermes 边界

- 用户管理面：创建/列出条目、创建 grant、撤销 grant；列表永不返回明文。
- Hermes：仅暴露 `agent_memory_use_protected_resource`；不能创建条目或 grant。
- 普通 `recall`、来源追溯和星图查询均不查询 `vault.entries.ciphertext`。
- API 日志只记录路径与状态，不记录请求/响应体；审计 metadata 只含 operation、target 和 ID。
- 无授权时 API 返回 `VAULT_GRANT_REQUIRED`；Provider 将其作为权限结果而非服务故障。

## 5. 已验证证据（2026-07-13）

- AES-GCM round-trip、entry ID 绑定、错误 ID 认证失败：单元测试通过。
- 超过 24 小时的 grant 被 schema 拒绝。
- API 集成覆盖：无 grant 拒绝、错误 profile 拒绝、显式 grant 允许、撤销后立即拒绝。
- Hermes 源码生命周期测试通过专用工具取得已授权值。
- 数据库密文字节扫描中测试明文标记命中 0。
- `pg_dump` 恢复到临时库后，使用独立根密钥实际解密恢复记录成功。

V1 测试 release 前仍需在星图 UI 中实现同样的人工治理流程、二次确认和脱敏关联展示。
