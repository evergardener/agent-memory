## 3. 为什么现在不能继续算法选型

当前数据只能证明三组真实的二元关系，无法形成三个可用于比较 Louvain、Leiden 或其他社区算法的
人工金标社区。如果把 receiver、alertname、状态词、代码类名或 assistant 推测补成第三颗行星，
虽然统计会“达标”，但会重新引入此前已经发现的不可读伪实体，并违反禁止推理生成虚假事实的
边界。

因此阶段 C 的正确阻塞点是代表性交互数据覆盖，不是算法或 UI。现有自动社区最低门槛保持不变。

## 4. 解除前置阻塞所需数据

优先方案是补充一份本地脱敏、排除 cron/自动通知注入的 Hermes 交互会话导出：

- 建议 20–50 个真实交互 session；
- 至少覆盖 3 个自然主题，例如家庭设备、监控服务、项目开发/排障；
- 每个主题应自然出现至少 3 个唯一实体；
- 每个候选关系至少有 2 条独立 user 消息、工具观察或可追溯事实支撑；
- 不需要为了满足数量人工编造对话，也不需要提供 Vault、密码、密钥或地址等敏感明文。

拿到导出后，仍先在本地生成候选审核清单和覆盖报告，达到金标条件后再进入社区算法比较与阶段 C
持久化设计。
