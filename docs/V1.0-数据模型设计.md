# Agent Memory for Hermes — V1.0 数据模型设计

## 1. 数据建模原则

1. **证据优先**：原始事件不可由整理流程覆盖；所有派生记忆可追溯到证据或人工操作。
2. **派生可重建**：事实以外的叙事、索引、向量、关系聚合、星图布局均可失效并重建。
3. **敏感隔离**：Vault 密文与普通记忆使用不同 schema；普通表只允许脱敏引用。
4. **版本而非覆写**：更正产生替代链与重建请求，不抹去原有历史；永久清除是唯一删除语义。
5. **一切可审计**：自动决策、用户治理、召回与 Vault 访问均记录操作主体、时间、目标和依据。

所有业务主键使用 UUID；时间为 `timestamptz`；正文采用 UTF-8 `text`；枚举先以受约束的 `text` 值实现，避免早期数据库迁移僵化。所有用户数据表均含 `namespace_id`（Vault 密文表可经关联取得）与 `created_at`/`updated_at`。

## 2. Schema 与核心表

### 2.1 `core`

| 表 | 关键字段 | 语义 |
| --- | --- | --- |
| `namespaces` | `id`, `stable_key`, `status` | 单用户的稳定共享命名空间；`stable_key` 唯一 |
| `sources` | `id`, `namespace_id`, `source_profile`, `source_instance` | Hermes profile/实例来源，不是存储隔离边界 |
| `sessions` | `id`, `namespace_id`, `source_id`, `external_session_id`, `started_at`, `ended_at` | 外部会话边界；`namespace + source + external_session_id` 唯一 |
| `turns` | `id`, `session_id`, `external_turn_id`, `sequence_no`, `status` | 回合归属与排序，供幂等事件关联 |

### 2.2 `evidence`

| 表 | 关键字段 | 不变量 |
| --- | --- | --- |
| `events` | `id`, `namespace_id`, `turn_id`, `event_type`, `redacted_payload`, `payload_hash`, `occurred_at`, `retention_class`, `ingest_key` | 仅追加；`ingest_key` 全局唯一；正文为脱敏副本 |
| `event_links` | `event_id`, `related_event_id`, `relation_type` | 表达消息、工具调用、结果、观察之间的关联 |
| `redaction_findings` | `event_id`, `kind`, `span_hash`, `action`, `rule_version` | 记录检测结果，不保存原始敏感值 |

`event_type` 至少包括 `user_message`、`assistant_message`、`tool_call`、`tool_result`、`environment_observation`、`session_boundary`。工具参数/推理摘要只能以脱敏、受策略控制的关联证据保存；不得默认保存模型私有推理链。

### 2.3 `memory`

| 表 | 关键字段 | 语义 |
| --- | --- | --- |
| `facts` | `id`, `namespace_id`, `fact_type`, `statement`, `confidence`, `valid_from`, `valid_to`, `memory_state`, `supersedes_fact_id`, `version` | 原子、可证实的陈述；长期/阶段/当前以 `fact_type` 区分 |
| `fact_evidence` | `fact_id`, `event_id`, `support_kind`, `weight` | 一个事实必须至少有一条 `support` 或人工更正依据 |
| `entities` | `id`, `namespace_id`, `entity_type`, `canonical_name`, `normalized_name`, `merge_state` | V1 以规范化名称自动合并项目/服务实体 |
| `fact_entities` | `fact_id`, `entity_id` | 事实与实体的证据关系 |
| `episodes` | `id`, `namespace_id`, `entity_id`, `title`, `summary`, `state`, `version` | 同一实体至少两个活跃阶段事实形成的保守聚合 |
| `episode_facts` | `episode_id`, `fact_id` | episode 的全部支撑事实；可继续回溯到 evidence |
| `arcs` | `id`, `namespace_id`, `entity_id`, `title`, `summary`, `state`, `version` | 同一实体至少两个活跃长期事实形成的保守脉络 |
| `arc_facts` | `arc_id`, `fact_id` | arc 的全部支撑事实；V1 不生成无证据叙事 |

### 2.4 `state`

| 表 | 关键字段 | 语义 |
| --- | --- | --- |
| `current_items` | `id`, `namespace_id`, `statement`, `status`, `expires_at`, `source_fact_id` | 天气、进行中事项等；到期后不主动召回 |
| `continuities` | `id`, `namespace_id`, `topic_key`, `summary`, `last_active_at`, `expires_at` | 跨会话短期连续性 |
| `interaction_snapshots` | `id`, `namespace_id`, `axes`, `calculated_at`, `algorithm_version` | jiwen 理念的五轴状态快照 |
| `settings` | `namespace_id`, `enabled`, `drift_hours`, `axes_initial`, `axis_labels`, `axis_ranges`, `axis_enabled`, `thresholds`, `profile_overrides` | 状态参数控制面；修改与重置进入 audit |

### 2.5 `retrieval`

`documents` 是事实、实体概览、episode、arc、状态的可丢弃检索投影。字段包括 `id`、`namespace_id`、`source_kind`、`source_id`、`text_redacted`、`tsv`、`embedding`、`embedding_model_version`、`lifecycle_state`、`indexed_at`。它不保存 Vault 明文，也不替代来源表。

### 2.6 `vault`、`ops` 与 `audit`

| schema | 表 | 关键字段 |
| --- | --- | --- |
| `vault` | `entries` | `id`, `namespace_id`, `kind`, `display_label`, `redacted_hint`, `ciphertext`, `wrapped_dek`, `key_version`, `status` |
| `vault` | `grants` | `id`, `entry_id`, `grantee`, `operation`, `target_constraint`, `expires_at`, `revoked_at` |
| `ops` | `jobs` | `id`, `kind`, `idempotency_key`, `input_version`, `status`, `run_after`, `attempt_count`, `payload_ref` |
| `ops` | `job_attempts` | `job_id`, `started_at`, `ended_at`, `result`, `error_code`, `correlation_id` |
| `audit` | `events` | `id`, `actor_type`, `actor_id`, `action`, `target_type`, `target_id`, `reason`, `correlation_id`, `metadata_redacted` |

Vault 条目在星图中的关联由普通 `memory` 表存储 `vault_entry_id` 的受限引用或专门的 `vault_references` 表表达，且只提供 `display_label`/`redacted_hint`。

## 3. 生命周期状态机

### 3.1 证据

`accepted → retained → purge_requested → purged`。`accepted` 和 `retained` 不可编辑；脱敏规则升级时可增加新投影，但不覆盖旧内容。`purged` 仅保留最小化、无正文的审计事实。

### 3.2 事实与关联

`candidate → active → dormant → forgotten` 是默认生命周期；`isolated` 表示用户删除关联后不参与召回；`superseded` 指向更正版本；`purge_requested → purged` 由永久清除工作流执行。`forgotten` 不主动召回，但命中明确主题时可作为低权重历史候选。任何自动生成记录不得直接进入 `active`，除非证据规则/置信度阈值满足。

### 3.3 派生叙事与投影

V1 采用事务内“删除旧投影并从当前有效事实重建”的保守策略。上游事实被更正、遗忘、孤立或清除时均入队 `rebuild_derived`；不足两个支撑事实的 episode/arc 自动撤销。后续若需要长时间模型整理，再引入显式 `stale/rebuilding` 状态。

## 4. 完整性与约束

- `facts`、`relations`、`episodes`、`arcs` 与其支撑对象必须位于同一 namespace。
- 自动事实的 `confidence`、抽取版本、模型配置版本和至少一个证据关联为必填；人工事实须有审计原因。
- `relations` 的两个实体不可相同，除非关系类型明确允许自引用。
- `current_items.expires_at` 必填且受最大 TTL 策略约束。
- 普通表与检索表禁止含 `vault.entries.ciphertext` 的可解密副本、敏感原文或根密钥。
- 用户操作“删除”仅改为 `isolated` 并断开关系；“忘记”降低召回状态；“更正”创建新版本；“永久清除”才触发物理删除/加密销毁。

## 5. 索引与分区基线

- 所有高频查询以 `(namespace_id, status/state, updated_at)` 为基础复合索引。
- `events.ingest_key`、`jobs.idempotency_key` 为唯一索引。
- `documents.tsv` 建 GIN FTS 索引；短名称和命令字段采用 `pg_trgm` 索引。
- 向量索引以真实数据基准后选用精确扫描或 HNSW；任何向量检索均先做 namespace/scope/lifecycle 过滤。
- 审计和证据在规模增长后按月分区，但 V1 初始迁移不应为尚未发生的规模提前复杂化。
