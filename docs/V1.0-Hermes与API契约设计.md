# Agent Memory for Hermes — V1.0 Hermes 与 API 契约设计

> 2026-07-21 实施修订：Subject 身份与来源映射、`planetary-v2` 及 `community-projection-v1` 已实现并通过隔离 Gate。本文新增的 `display_name_origin` 和纯人格名响应规则为阶段 E0 目标契约，尚未实现，完成前不得标记为已验证。

## 1. 契约目标

该契约隔离 Hermes Provider、API、worker 与星图 UI。Provider 不理解数据库结构；UI 不直接访问数据库；所有外部输入经过 API 验证、授权、审计与事务处理。

版本前缀为 `/api/v1`。所有请求与响应包含 `correlation_id`；变更型请求还必须包含稳定 `idempotency_key`。API 返回的记忆正文均为普通记忆的脱敏投影，绝不包含 Vault 明文。

## 2. 身份、范围与错误

### 2.1 Provider 请求上下文

每次 Provider 请求须携带：

```json
{
  "shared_namespace": "hermes:user-primary",
  "source_profile": "default",
  "source_instance": "local-tui",
  "external_session_id": "hermes-session-id",
  "external_turn_id": "turn-id",
  "correlation_id": "uuid"
}
```

API 使用安装时发放的 Bearer token 验证 Provider。`shared_namespace` 必须是 Provider 安装配置允许的值；`source_profile` 只作来源记录、profile 人格主体解析和可选筛选，不得形成新的物理记忆隔离。`source_instance` 和 session 不能自动生成星图恒星；同一 profile 的多个实例必须解析到同一稳定 Subject，除非用户显式配置为不同人格。

### 2.2 错误模型

```json
{
  "error": {
    "code": "VAULT_GRANT_REQUIRED",
    "message": "该受保护资源需要用户明确授权",
    "retryable": false,
    "correlation_id": "uuid"
  }
}
```

标准错误码：`UNAUTHENTICATED`、`NAMESPACE_DENIED`、`VALIDATION_ERROR`、`IDEMPOTENCY_CONFLICT`、`NOT_FOUND`、`CONFLICT`、`VAULT_GRANT_REQUIRED`、`VAULT_ACCESS_DENIED`、`SERVICE_UNAVAILABLE`。Provider 对 `SERVICE_UNAVAILABLE` 可退避重试；对其他错误不得盲目重试。

## 3. Hermes Provider 生命周期映射

| Hermes 生命周期 | Provider 行为 | API 行为 |
| --- | --- | --- |
| `initialize` | 加载本地配置、令牌、namespace、session 和 profile；不建库 | 解析已配置的 profile 人格 Subject；未知映射进入待治理状态，不用实例 ID 创建恒星 |
| `on_turn_start` / `prefetch` | 发送查询与预算，请求上下文 | `POST /api/v1/recall` 返回可注入片段 |
| `sync_turn` | 归一化消息、工具调用、工具结果并提交一个回合 | `POST /api/v1/ingest/turn` 原子写入证据和任务 |
| `on_session_switch` | 更新本地 session，清零 turn 序号 | 下一次 ingest 使用新 `external_session_id` |
| `on_pre_compress` | 获取最多三条紧凑连续性摘要 | `GET /api/v1/state`，不产生新事实 |
| `handle_tool_call` | 将 Hermes memory tools 路由至 API | 见第 5 节 |

若 API 不可用，Provider 必须返回空的附加上下文而非中断 Hermes；可使用内存中的、短 TTL、无敏感内容的最近召回缓存。缓存不能用于写入确认，也不能跨 profile/namespace 复用。

## 4. Provider 核心端点

### 4.1 `POST /api/v1/recall`

请求：

```json
{
  "context": {"shared_namespace": "hermes:user-primary", "source_profile": "default", "external_session_id": "s1", "external_turn_id": "t18", "correlation_id": "uuid"},
  "query": "继续之前的 agent-memory 数据库设计",
  "intent": "conversation",
  "budget": {"max_items": 8, "max_chars": 4200},
  "scopes": ["global", "project", "phase"]
}
```

响应：

```json
{
  "items": [
    {
      "memory_id": "uuid",
      "kind": "fact",
      "text": "项目采用 PostgreSQL 与 pgvector 作为统一数据核心。",
      "source_ids": ["evidence-uuid"],
      "channels": ["lexical", "relation"],
      "why_recalled": "项目实体与查询词匹配",
      "permission": "recall"
    }
  ],
  "truncated": false,
  "correlation_id": "uuid"
}
```

响应项必须经 lifecycle、scope、用户治理及 Vault 策略过滤。`why_recalled` 只说明匹配原因，不泄露被过滤的记忆。

### 4.2 `POST /api/v1/ingest/turn`

请求包含 Provider 上下文、`idempotency_key`、发生时间及按顺序的事件数组：

```json
{
  "context": {"shared_namespace": "hermes:user-primary", "source_profile": "default", "source_instance": "local-tui", "external_session_id": "s1", "external_turn_id": "t18", "correlation_id": "uuid"},
  "idempotency_key": "sha256:stable-turn-key",
  "occurred_at": "2026-07-13T10:00:00Z",
  "events": [
    {"type": "user_message", "sequence": 1, "content": "..."},
    {"type": "assistant_message", "sequence": 2, "content": "..."},
    {"type": "tool_call", "sequence": 3, "tool_name": "...", "arguments": {}},
    {"type": "tool_result", "sequence": 4, "tool_name": "...", "content": "..."}
  ]
}
```

API 先脱敏再持久化；响应仅确认事件 ID、是否重复与已入队任务 ID。成功确认后证据已持久化，但不代表事实已抽取。推理链不属于默认 `events`；如确需保留，只能写入单独受策略约束的摘要/关联记录，默认关闭。

### 4.3 会话与连续性

V1 Provider 的 `on_session_switch` 更新本地 session 上下文；回合写入的 `external_session_id` 在服务端形成边界。`on_pre_compress` 通过 `GET /api/v1/state` 读取最多三条有证据的连续性摘要，不新增事实，也不使用独立的模型压缩端点。

## 5. Hermes 记忆工具

| Hermes 工具 | API | 规则 |
| --- | --- | --- |
| `agent_memory_recall` | `POST /api/v1/recall` | 使用 `explicit` 意图；允许强主题命中唤醒 forgotten，仍排除 isolated/purged |
| `agent_memory_trace_source` | `GET /api/v1/memory/{id}/trace` | 对 fact/episode/arc 返回脱敏证据链与版本关系 |
| `agent_memory_current_state` | `GET /api/v1/state` | 读取确定性状态、当前事实和连续性 |
| `agent_memory_update_current_state` | `POST /api/v1/state/items` | `set`/`update`/`resolve`；设置与更新必须带 TTL |
| `agent_memory_correct` | `POST /api/v1/memory/{id}/corrections` | 创建替代版本并入队派生层重建 |
| `agent_memory_use_protected_resource` | `POST /api/v1/vault/requests` | 仅在有效 profile grant 下执行 `reveal_to_model` |

`request_protected_resource` 的默认结果是授权状态或代理执行的脱敏结果。它不应成为“把密钥注入模型上下文”的通道。

## 6. 星图与治理端点

UI 使用独立登录会话；不复用 Provider token。

- `GET /api/v1/graph/subgraph`：`planetary-v2` 返回主体恒星/实体行星与非天体 overlay；`view=universe|galaxy` 和可选 `galaxy_id` 启用 `community-projection-v1`。主宇宙返回可见星系入口和类型关系，子星系只返回成员行星与内部关系。`profile`、`fact_type`、`lifecycle`、`activity`、`sensitivity` 及 `updated_after` 镜片只过滤 overlay/关系，不改变 canonical entity、成员或天体稳定 ID。
- `GET /api/v1/graph/subjects`：返回用户和 profile 人格主体、可读显示名、名称来源、颜色/状态及脱敏来源映射；同 profile 多实例聚合为一个 Subject。视觉标题只使用 `display_name`，`kind`/`source_profile` 为详情字段，不由客户端拼回标题。
- `PUT /api/v1/graph/subjects/{id}`、`POST /api/v1/graph/subjects/{id}/sources/{source_id}`、`DELETE /api/v1/graph/subjects/{id}/sources/{source_id}`：人工修改主体显示、建立映射或撤销人工映射并恢复基于 profile 的默认映射；均须 UI 会话、原因和审计，不能修改原始 evidence 来源字段。人工改名须将 `display_name_origin` 设为 `manual`，后台同步不得覆盖。
- `GET /api/v1/graph/galaxies`：返回关系社区、成员数、自动/人工来源、算法版本、输入快照和可解释成员摘要；自动结果不得包含单体星系。
- `POST /api/v1/graph/galaxies`、`PATCH /api/v1/graph/galaxies/{id}`：人工创建、命名、锁定或隐藏星系，使用版本号做乐观并发控制。
- `PUT /api/v1/graph/galaxies/{id}/members/{entity_id}`：固定、恢复自动、设为主/次成员或排除行星；操作只表达信息架构治理，不创建现实事实。
- `POST /api/v1/graph/galaxies/rebuild`：按 namespace 入队重算自动社区，保留人工命名、固定、排除和位置偏好。
- `POST /api/v1/graph/galaxies/{id}/undo`：撤销最近一次可逆的星系/成员治理操作，仍执行版本冲突检查并写审计。
- `GET|PUT /api/v1/graph/layout`：读取或保存星系/行星位置、动态开关和用户固定；布局数据不得参与事实召回。
- `GET /api/v1/memory/{id}/trace`：返回 fact/episode/arc 的证据链和版本关系。
- `GET /api/v1/memories/review`：独立的治理队列，不受星图子图条数限制。必须指定当前
  `shared_namespace`，支持 `reason=all|candidate|untrusted_tool`、`source_profile`、
  `limit`（1–200）和 `offset`；响应包含总数、当前页、可筛选 profile 和脱敏后的事实摘要。
  排序固定为不可信工具优先，其后按更新时间和 ID 倒序，以保证翻页稳定。
- `POST /api/v1/memory/{id}/isolate|forget|corrections|purge`：分别执行脱离、忘记、更正和受控清除。
- `POST /api/v1/entities/{id}/merge`：把源实体可逆地解析到目标规范实体；不删除源实体、事实或证据。
- `POST /api/v1/entities/{id}/unmerge`：撤销合并并恢复源实体的原始事实关联。
- `POST /api/v1/entities/{id}/split`：只把用户明确选择的事实关联移动到新实体；原始证据保持只读。
- `POST /api/v1/entities/{id}/facts/{fact_id}/attach|detach`：人工建立或解除实体—事实关系；保留事实与原始证据，写入审计并重建派生层。
- `GET|PUT /api/v1/state`、`/api/v1/state/config`：按解析后的 profile 人格 Subject 读取状态，并治理 namespace 默认名称、范围、初始值、启停、漂移、阈值和 Subject override；状态分轨不得改变共享事实范围。
- `POST /api/v1/state/simulate|reset|items`：无副作用模拟、受审计重置和当前事实管理。
- `GET /api/v1/reports/consolidation`：查看整理报告。
- `GET /api/v1/reports/quality`：返回只含聚合指标的导入质量门禁，不返回对话正文。自动门禁即使通过也只进入人工抽检阶段，API 不会自动批准进入主 namespace。
- `/api/v1/vault/entries`、`grants`：只接受 UI 会话。支持创建、脱敏列表、元数据修改、人工明文查看、敏感值替换、停用/启用、永久删除、限时授权和撤销；查看、修改、替换、状态变更与删除必须再次验证本地管理密码。明文查看响应设置 `no-store`，替换/停用会撤销现有授权，永久删除物理移除密文和授权且只保留无秘密审计事件。
- `POST /api/v1/vault/requests`：只接受 Provider service token；还必须命中未过期、未撤销、指定 profile 和操作的 grant。UI 会话不能调用该模型读取端点，Provider token 也不能调用 Vault 管理端点。

图端点响应必须遵守下列投影契约：

- `node_kind=subject` 显示为恒星，`node_kind=entity` 显示为行星；Subject 引用的 entity 不得重复返回为行星；
- fact/episode/arc/report/source/session/job 不能作为天体节点；episode/arc 只能作为可追溯的星座/星流 overlay；
- 星系成员是多对多投影，`is_primary` 只表示布局主位置；
- 社区名称、权重和自动摘要必须标记 `origin`、`algorithm_version` 和解释引用；
- Vault 只返回保护标记、条目 ID 和脱敏提示，不返回密文或明文。

## 7. 异步任务契约

API 写入任务时只传递对象引用和版本，不在 job payload 复制完整敏感正文。V1 任务类型为 `extract_facts`、`rebuild_derived`、`rebuild_communities`、`generate_report`、`purge_memory`；抽取任务在同一事务生成实体关联、检索投影、连续性与状态快照。

Worker 获取任务必须采用数据库锁定/租约机制；每次尝试写入 `job_attempts`。完成任务前须确认输入版本仍然有效，否则取消旧任务并按新版本重排。用户的更正、隔离和清除优先于正在执行的整理任务。

## 8. 契约验收测试

1. 相同 `ingest/turn` 请求重复提交只产生一份证据和一组任务。
2. 两个 source profile 在同一 namespace 可相互召回，`trace` 可显示来源 profile。
3. `recall` 返回项均可追溯到证据或人工审计依据。
4. API/worker/模型任一不可用时，Hermes 回合不失败且不虚构记忆。
5. 伪造 Token 写入后，`recall`、`graph`、日志和模型模拟请求中均没有明文。
6. 无 Vault grant 的 Provider 请求被拒绝；撤销后立即失效并产生审计。
7. 治理队列跨页无重复，不能跨 namespace 读取，且候选/不可信工具/profile 筛选不依赖星图是否加载该事实。
8. 实体合并后目标可解析源实体的事实，撤销后关联恢复；拆分只移动选定事实。三种操作均有审计且不能跨 namespace。
9. 用户与两个 profile 人格返回三个稳定 Subject；同 profile 的第二个实例不增加 Subject 数量，未知实例 ID 不成为实体名称。
10. 普通 person/project/device/service/location 返回为唯一行星；Subject 引用的 entity 不重复返回为行星。
11. 同一行星可出现在多个 galaxy membership 中，但治理、追溯和 canonical entity ID 相同；fact_type 镜片切换不改变 membership。
12. 自动社区不足三个行星或缺少独立行星关系时不生成星系；人工创建例外须明确 `origin=manual`。
13. 星系重建后人工名称、成员固定/排除和布局偏好保持不变；自动变化可审计并可撤销。
