# Agent Memory V1.0 生产来源治理与部署冻结设计

> 适用版本：`1.0.0-rc.8`。本文定义多 profile 恒星、生产来源边界、运行版本冻结和备份新鲜度。它不授权修改当前 Hermes、生产数据或容器。

## 1. 目标与非目标

目标：

- 单用户的多个 Hermes profile 使用同一 namespace 和记忆空间，但每个 profile 形成独立人格恒星；
- 同 profile 的多个实例、TUI 和会话归并到同一恒星，同时保留实际来源；
- canary 必须证明每个获批 live profile/instance 自身写入了 evidence；
- 历史导入可以与 live 数据共存，但不能用全局 event 数替代 live canary 门禁；
- 生产操作必须绑定已验证的源码、运维脚本和镜像，而不是隐式信任可变 `main`；
- 备份状态明确区分 canary 前和 canary 后覆盖范围。

非目标：

- 不把 profile 变成新的租户或数据库隔离边界；
- 不按 profile 自动删除、移动或重写既有 evidence；
- 不从 Hermes 私有配置猜测人格合并关系；
- 不因源码更新自动 rebuild、restart、checkout 或修改生产配置。

## 2. 身份、来源与角色

三个概念必须分开：

| 概念 | 稳定键 | 含义 |
| --- | --- | --- |
| 共享记忆空间 | `namespace` | 单用户的事实、实体、情节、脉络和 Vault 治理边界 |
| 人格恒星 | `profile:<casefold-profile>` | 一个用户可理解、可改名的 Hermes profile 主体 |
| 实际来源 | `(source_profile, source_instance)` | evidence 的真实接入路径和审计身份 |

默认映射规则：

1. 不同 `source_profile` 映射为不同 profile 恒星；
2. 同一 `source_profile` 的多个 `source_instance` 映射为同一恒星；
3. 人工 Subject 映射可纠正例外，但不修改原始 evidence 来源字段；
4. source instance、内部 ID、时间戳和历史导入批次都不成为星体名称。

生产来源策略 `SOURCE-POLICY.json` 只允许两种角色：

- `live_profile`：当前真实 Hermes 接入；必须用自己的 evidence 通过 canary，独立计算观察窗口；
- `historical_import`：用户明确批准的只读历史来源；允许保留和参与记忆，但永远不能满足 live gate。

新增或改变来源角色是生产治理动作，必须使用明确确认短语并更新状态中的策略指纹。未知来源默认失败关闭。

## 3. 多 profile canary 状态机

每个获批 live source 在 `canary_sources` 中保存：

- `source_profile`、`source_instance`、`role`；
- Hermes env 文件路径及 SHA-256，不保存其中的 token；
- `configured_at`；
- `verification_status=prepared|verified`；
- 自身 event count、最近验证时间和 `first_verified_at`。

验证必须同时满足：

1. source policy 中恰有预期的 `live_profile` 项；
2. inventory 中存在相同 profile + instance；
3. 该 source 自身至少有一条 event；
4. inventory 中没有未批准来源；
5. `failed_jobs=0`；
6. 输出和状态只包含来源身份、计数、时间范围及 inventory hash，不含 payload 或 session ID。

多个 profile 可逐个加入。新 profile 不重置旧 profile 的稳定身份，但它自己的 72 小时观察从首次 source-bound 验证开始。晋级时以所有 live source 中最晚达到的观察窗口为准，不能继承第一个 profile 已累计的时长。

## 4. 只读来源 inventory

生产只读报告命令：

```bash
bash scripts/production-source-inventory.sh \
  "$runtime_root/production.env"

bash scripts/production-source-inventory.sh \
  "$runtime_root/production.env" --json \
  "$runtime_root/SOURCE-INVENTORY.json"
```

报告允许包含：

- profile、instance、策略角色和批准状态；
- session、turn、event 和 evidence-linked fact 数；
- first/last event timestamp；
- namespace 级 Vault entry 总数和按目标 profile 聚合的 active grant 数；
- inventory 与 policy SHA-256。

报告禁止包含 evidence payload、对话正文、session ID、Vault ciphertext/plaintext、token、密码或环境文件内容。Vault 无可靠 source instance 归属，因此只报告 namespace 总量和 grant 目标，不伪造“每来源 Vault 数”。

## 5. 运行版本冻结

首次部署后在运行目录生成只读 deployment bundle。清单绑定：

- Git full revision 与 `VERSION`；
- `compose.yaml`、`compose.production.yaml` 和关键生产脚本的 SHA-256；
- API、Worker、Migrate 镜像 ID 与 OCI revision；
- bundle 绝对路径和清单 SHA-256。

`production-up/verify/backup/stop/promote` 等生产入口先检查：

1. env/state 的 version、revision、project 和 namespace 一致；
2. 当前关键文件与 bundle 副本均匹配清单；
3. 当前镜像 tag 的 image ID 和 OCI revision 匹配清单；
4. source policy 的规范化 SHA 与状态一致。

任何不一致都失败关闭，只输出文件名、revision、路径或 hash，不自动修复。纯文档变更不在关键文件列表中，因此不会影响已绑定运行操作；改动 Compose 或生产脚本一定会触发失败。

## 6. 备份一致性与新鲜度

生产备份流程：

1. 先完成 runtime 只读验证；
2. 暂停 API、core worker，以及已批准启用的 model worker；
3. 执行 PostgreSQL custom-format dump；
4. 复制 Compose、运行 env、版本锁、状态、来源策略和部署清单；
5. 校验全部 SHA-256；
6. 在独立临时数据库恢复，比较关键表计数并验证 Vault 解密；
7. 原子更新 backup state；
8. 无论成功或失败都尝试恢复已暂停 writer。

新鲜度状态：

- `pre_canary_ready`：canary 尚未开始，备份在允许年龄内；
- `pre_canary_only`：canary 已开始，但最新恢复验证备份早于开始时间；
- `post_canary`：最新恢复验证备份覆盖 canary 数据。

`runtime` 只报告状态；`canary` 默认拒绝 `pre_canary_only`。尚处观察期且明确接受该风险时，可使用：

```bash
bash scripts/production-verify.sh \
  "$runtime_root/production.env" canary existing '<profile>' \
  --allow-pre-canary-backup-for-observation
```

该参数不创建备份，也不能用于晋级。晋级先创建并恢复验证新备份，再重新执行所有 profile 的 source-bound canary Gate。

## 7. 人工治理路径

发现既有来源时只允许三种人工决策：

1. **保留**：以 `historical_import` 或 `live_profile` 明确加入策略；角色必须反映真实用途；
2. **隔离/休眠**：通过已有事实、实体和记忆治理链路降低召回，不改写只读 evidence；
3. **重新初始化**：仅在用户批准、备份恢复通过并确认舍弃当前运行资产后创建新生产目录。

策略命令示例：

```bash
bash scripts/production-source-policy.sh \
  "$runtime_root/production.env" \
  '<profile>' '<instance>' historical_import \
  APPROVE_PRODUCTION_SOURCE_POLICY
```

项目不提供按 profile 批量 delete/purge 命令。策略变更只决定来源是否获批，不删除数据，也不改变恒星人工名称。

## 8. 旧运行态升级与重新绑定

rc.8 不会把缺少 deployment manifest/source policy 的旧运行态静默认领为可信运行态。旧运行态执行新生产脚本会失败关闭，这是预期行为。

升级必须作为单独批准的变更窗口执行：

1. 从状态和运行容器只读确认旧 revision、镜像 ID、namespace、来源 inventory；
2. 从旧 revision 的固定 checkout 使用旧版脚本创建并恢复验证升级前备份；
3. 人工确定每个既有来源的 `live_profile|historical_import` 角色；
4. 在隔离 release runtime 对 rc.8 完成完整 Gate；
5. 向用户提交升级 revision、镜像 ID、来源策略、备份和回滚点并等待批准；
6. 批准后才允许更新生产 env/state、重建容器并生成新的只读 deployment bundle；
7. 更新后立即验证 API、Worker、迁移、来源 inventory、星图、备份恢复和 Hermes fail-soft；
8. 任一验证失败按已批准回滚方案处理，不清库、不自动降级数据库。

当前开发任务只交付代码、测试和操作设计，不执行上述升级，不修改 Hermes profile，不重启当前容器。

## 9. 验收矩阵

| 场景 | 预期 |
| --- | --- |
| 预期 live source 无 event，其他来源有 event | 失败 |
| 未批准来源出现 | 失败并只列 profile/instance |
| 历史来源已批准，live source 自身有 event | 通过 |
| 历史来源有 event，live source 无 event | 失败 |
| profile/instance 含非法字符 | 失败 |
| 同 profile 多 instance | 同一恒星，来源分别审计 |
| 新增第二 profile | 新恒星，独立首次验证与观察窗口 |
| 关键脚本或 Compose 改变 | 生产操作失败 |
| 镜像 tag 指向不同 image ID | 生产操作失败 |
| canary 前备份用于常规 canary 验证 | 默认失败；观察例外需显式参数 |
| 晋级前无 post-canary 备份 | 失败 |
| 备份中途失败 | writer 恢复尝试，状态不标记为已验证 |
