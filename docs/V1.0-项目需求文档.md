# Agent Memory for Hermes — V1.0 项目需求文档

## 1. 文档目的

定义面向 Hermes 的本地长期记忆系统 V1.0 产品需求，作为后续架构、技术选型、实现和验收的约束基线。本文件不指定编程语言、数据库、模型厂商或部署编排方案。

系统以 Memory Constellations 的“提取、整理、召回、星图治理”理念为基础，并整合 jiwen 的“连续、确定性、可模拟的互动状态机”理念；系统强化证据可追溯性、长期可靠性、人工纠错、敏感信息隔离和 Hermes 多 profile 连续性。

## 2. 产品目标与范围

### 2.1 目标

1. 让 Hermes 跨会话、跨 profile、跨实例稳定记住用户、项目、服务环境、经历和决策。
2. 让 Hermes 在恰当的时机召回相关历史，而非向上下文注入大量无关旧内容。
3. 让用户通过星图理解记忆、追溯来源、修正错误、控制遗忘和管理敏感资源。
4. 在长期运行中保持证据完整、派生记忆可重建、自动整理可审计、敏感数据不泄露。

### 2.2 V1 范围

- 单用户、单 Hermes 身份。
- Hermes 多 profile、会话和实例共享统一记忆空间。
- Docker 化本地部署；普通数据和处理默认在本地。
- 支持外部模型 API 与本地部署模型；调用端点（含自定义 base URL）可配置。
- 支持原始证据、事实、实体关系、情节、长期脉络、当前状态和会话连续性。
- 支持星图浏览、追溯、编辑、遗忘、更正、永久清除和整理报告。
- 支持独立敏感信息库与凭据管理，并从星图 UI 统一进入。
- 支持独立、可配置、可解释的连续互动状态模块；默认不执行自主外发行为。

### 2.3 V1 非目标

- 多用户、多租户及多独立 Agent 的权限/冲突模型。
- 云端同步与第三方 Agent 的通用接入框架。
- 未经用户显式启用和授权的自主外发消息、主动通知或状态驱动的外部操作。
- 无证据的自动推断、将模型推理链写入记忆、自动永久删除内容。

## 3. 总体业务框架

```text
Hermes 多 profile / 实例
          ↓
  原始证据（只读、可追溯、敏感值脱敏）
          ↓
 Scribe：候选事实、实体、关系提取
          ↓
 Archivist：轻量维护 / 深度整合 / 生命周期管理
          ↓
 Librarian：轻量自动召回 / 主题召回 / 深度检索
          ↓
Hermes 上下文与显式记忆工具

独立状态模块：时间漂移 + 受证据约束的状态变化 → 互动/语气建议
  └─ 不修改事实记忆；不直接执行外部操作

星图 UI：浏览、追溯、编辑、整理报告、受保护资源入口
```

## 4. 核心原则

1. **证据优先**：任何稳定事实、情节或长期脉络必须能回溯到来源证据。
2. **不虚构**：自动处理只能提取、压缩、归纳和链接既有证据；不得补全未知细节。
3. **证据不可改，派生可重建**：原始证据只读；派生内容可因更正、删除或重新整理而重建。
4. **默认自动，始终可治理**：后台默认执行整理；用户可在星图中检查、修正、撤销和清除。
5. **按需召回**：日常只注入极少量高相关内容；历史细节在明确主题或显式检索时展开。
6. **敏感隔离**：敏感明文不进入普通证据、普通记忆、星图节点、整理报告或模型上下文。
7. **本地优先、模型可替换**：外部或本地模型均可用，但敏感信息不会默认发送到任一模型。
8. **状态与事实解耦**：连续互动状态用于表达“现在如何回应/是否值得提醒”，不是关于用户或世界的事实。
9. **状态可复演**：相同的初始状态、事件与时间输入必须产生相同状态和建议；状态参数修改前应可模拟评估。

## 5. 证据与事实准入

### 5.1 事件来源

| 来源 | 是否保存为证据 | 是否可形成候选事实 | 规则 |
| --- | --- | --- | --- |
| 用户消息 | 是 | 是 | 用户偏好、经历、决策、明确陈述与纠正可候选化 |
| Hermes 回复 | 是 | 谨慎 | 仅在得到用户确认、工具验证或其他证据支持后才能成为事实 |
| 工具调用结果 | 是 | 是 | 必须带工具、时间、结果摘要、来源和时效 |
| 环境观察 | 是 | 是 | 必须记录观测来源、时间、范围和可信度 |
| 工具参数/运行记录 | 可关联保存 | 否 | 作为审计资料；密钥、Token 等须过滤 |
| 模型推理链 | 否 | 否 | 不作为可靠事实或知识来源 |
| 用户修正 | 是 | 是 | 形成更正记录并优先于被替代内容 |

### 5.2 认知状态

- 用户陈述：用户明确表达的信息。
- 工具观测：由工具或环境明确观察到的信息。
- 多证据归纳：多个来源支持的压缩结论。
- 用户修正：用户主动更新或否定旧内容。
- Agent 推测：只作为候选关联，不可自动成为稳定事实。
- 派生摘要：基于事实生成的可读描述，必须标示为摘要。

### 5.3 默认分流

“默认优先收入记忆”定义为：默认捕获原始证据并积极生成候选项；只有通过分类、证据和生命周期规则后，才进入可主动召回的正式记忆层。

模糊项默认创建候选记忆并进入整理报告；不要求用户逐条即时确认。敏感内容例外：先脱敏或阻断，再决定是否由用户手动保存到敏感信息库。

## 6. 记忆模型

| 层级 | 内容 | 例子 | 默认召回行为 |
| --- | --- | --- | --- |
| 原始证据 | 会话、工具结果、环境观察、编辑历史 | 原始对话、命令结果 | 不直接注入；可追溯 |
| 长期事实 | 稳定或缓慢变化的信息 | 服务拓扑、部署职责、偏好、决策 | 高相关时主动召回 |
| 阶段事实 | 与一个时期或主题关联的信息 | 项目开发、旅行、排障 | 活跃时主动；后期按主题召回 |
| 实体与关系 | 人、profile、项目、服务、工具、概念及关联 | 服务依赖、项目归属 | 支撑联想与星图 |
| 情节 | 足够证据支持的阶段叙事 | 某次故障及修复过程 | 主题匹配时召回 |
| 长期脉络 | 跨情节的长期演变 | 项目进展、反复问题 | 低频、按需召回 |
| 当前状态 | 暂时有效并影响当前回应的事实 | 今日天气、服务健康、临时安排 | 高优先、到期失活 |
| 会话连续性 | 最近目标、待办、未解决问题 | 当前任务摘要 | 短时优先，跨 profile 共享 |

同一事件可以产生多个层级的派生内容。例如，旅行期间的对话可形成阶段事实；其中的具体天气为当前状态；旅行中的长期偏好可形成长期事实。

## 7. 具体分类边界

### 7.1 长期事实

- 内网服务拓扑、稳定部署情况、服务职责、重要访问入口。
- 用户非敏感资料、偏好、工作习惯和常用环境。
- 用户或 Hermes 已做出的重要决策及其依据。

运行健康、临时 IP、短期告警等不属于稳定部署事实，应作为当前状态或阶段事实处理。

### 7.2 阶段事实

- 与 Hermes 协作开发项目的进展、决策、问题和结果。
- 外出旅行期间的计划、对话背景、事件和体验。
- 排障过程、变更过程、迁移过程及其阶段性结论。

### 7.3 当前状态与低价值信息

- “今天上海有暴雨预警”是当前状态，具有明确时间范围。
- “天气怎么样”这个问询本身属于低价值事件。
- “某 Linux 命令怎么用”通常仅保留为原始证据。
- “已修改某服务的配置并验证生效”是阶段事实，必要时更新长期服务知识。

### 7.4 敏感信息

- 身份证号、详细住址、联系方式等高敏感个人资料。
- 密码、密钥、Token、Cookie、私钥、访问凭据等秘密。

敏感值默认脱敏保存于会话证据；完整值只能由用户手动保存到敏感信息库。

## 8. 自动整理

### 8.1 即时处理

每个事件进入后，系统立即保存可保留的原始证据，附加 profile、会话、时间、来源和关联信息。发现敏感值时先脱敏。

### 8.2 轻量整理

默认高频执行，不生成大段叙事：

- 事实去重与候选归类；
- 实体匹配、别名处理与关系更新；
- 当前状态失效；
- 低价值信息标记；
- 冲突识别；
- 可疑实体合并的证据积累。

### 8.3 深度整理

默认在会话边界清晰或系统空闲时执行：

- 将充分关联的事实合并为情节；
- 由多个情节形成或更新长期脉络；
- 阶段记忆降级为休眠/忘记；
- 因更正或清除而重建派生内容；
- 生成可查看的整理报告。

实体可自动合并，但必须记录合并依据、置信度、合并前后节点，并允许用户一键拆分或撤销。

### 8.4 整理触发策略

所有触发阈值均须可配置。系统必须支持以下触发方式：

- 新事件到达时的即时证据捕获；
- 会话进入空闲后的阶段整理；
- 未整理事件达到积压阈值后的批量整理；
- 周期性的轻量维护与低频深度整理；
- 用户从星图手动发起的指定范围整理或重建。

触发机制不得依赖某一固定分钟数；每次整理应记录触发原因、覆盖范围、输入版本、结果和失败/重试状态。

## 9. 生命周期与默认配置

所有时长均须可配置。以下为 V1 默认值：

| 对象 | 默认规则 |
| --- | --- |
| 普通原始证据 | 本地长期只读保存，直到用户永久清除 |
| 候选记忆 | 30 天未升级则停止参与整理，仅保留来源证据 |
| 当前状态 | 默认 7 天；天气/短期环境观察默认 24 小时；允许事件单独指定 |
| 会话连续性 | 默认 14 天；新会话可覆盖或压缩旧摘要 |
| 阶段事实/情节 | 90 天无相关活动进入休眠；365 天后进入忘记状态 |
| 忘记状态 | 不主动召回；仅在明确主题、实体命中或深度检索时出现 |
| 长期事实 | 保持活跃；允许定期复核、被更正或显式忘记 |
| 敏感条目 | 保留至用户禁用、替换或永久清除；秘密允许记录轮换状态 |

### 9.1 用户操作语义

| 操作 | 结果 |
| --- | --- |
| 脱离 | 断开关联，内容成为孤点且不参与正常召回 |
| 忘记 | 保留内容和证据，降低关联与主动召回权重 |
| 更正 | 写入用户修正，旧内容标为被替代，派生内容重建 |
| 永久清除 | 清除证据、派生内容、索引、星图节点和关联，不可恢复 |

## 10. 召回与上下文

### 10.0 召回能力基线

系统必须从至少三类独立候选来源中生成召回结果：

1. **词法候选**：对明确名称、命令、服务、项目、日期和关键词的精确或近似匹配。
2. **语义候选**：对相近表达、隐含主题和历史叙事的相关性匹配。
3. **实体关系候选**：通过人物、profile、项目、服务、工具、事件及其关系图进行联想。

系统必须能够将候选融合、去重、排序，并在结果中提供命中通道、来源证据、记忆状态和排序解释。具体检索引擎、索引格式和融合算法不在本需求文档中锁定。

### 10.1 每轮轻量召回

仅返回少量高置信、高相关内容：当前状态、直接相关的长期事实、最近会话连续性、当前项目的活跃阶段记忆。

### 10.2 主题触发召回

当用户提及明确人物、项目、服务、工具、地点或历史事件时，扩展到关联实体、阶段事实、情节和时间线。

### 10.3 深度历史检索

当用户要求回顾、Hermes 显式检索或需要复盘时，返回更完整的历史、来源、更正版本与冲突；内容应分批浏览，不应无预算地注入对话上下文。

### 10.4 排序原则

排序应综合明确实体匹配、当前任务相关性、证据可信度、用户修正、时效性、记忆状态与关系强度。被忘记和孤点内容不参与普通召回。

### 10.5 Hermes 记忆工具契约

Hermes 必须通过稳定的记忆接口访问本系统。接口名称可调整，但 V1 必须具备下列语义：

| 能力 | 最小输入 | 最小输出/行为 |
| --- | --- | --- |
| 召回记忆 | 查询、profile/范围、深度、上下文预算 | 排序后的事实/情节/实体/状态；命中通道、来源、置信度、记忆状态 |
| 浏览记忆 | 实体、类别、时间、状态或项目过滤 | 可用于星图或对话的图谱切片、节点摘要和关联 |
| 追溯来源 | 记忆或情节引用 | 脱敏后的原始证据、时间、来源类型和更正历史 |
| 更新当前状态 | 设置、更新或解决；有效期、范围、来源 | 创建/更新/结束短时状态，保留审计与失效规则 |
| 更正记忆 | 目标记忆、用户修正、原因 | 写入高优先级修正，标记旧版本并触发派生内容重建 |

所有接口必须遵循 profile/作用域过滤、敏感隔离和审计要求；任何接口均不得返回未授权的敏感明文。

## 11. 星图与用户治理

星图是 V1 的关键交付物和用户治理中心，不只是可视化效果。

### 11.1 页面与能力

- 总览：人物、项目、地点、事件、兴趣/知识等星系；用户与 Hermes 为核心节点。
- 过滤：按 profile、时间、类别、活跃度、敏感性和记忆状态筛选。
- 节点详情：事实、情节、长期脉络、当前状态、关系、时间线和来源证据。
- 状态面板：以可读描述展示 Hermes 当前互动状态、近期变化、触发原因和已产生的建议；不默认展示或向模型注入裸数值。
- 关系治理：查看关联依据、调整关系、合并/拆分实体、撤销自动合并。
- 记忆治理：更正、脱离、忘记、永久清除、标记敏感。
- 整理报告：查看新增、合并、冲突、过期、休眠、忘记和待确认项。

### 11.2 整理报告

默认每 7 天生成一次，保存在星图中供用户手动查看。后续 Hermes 接入通知渠道后，可将报告摘要推送到用户指定渠道。

## 12. 敏感信息库与凭据管理

敏感信息库从星图 UI 的“受保护资源”一级入口进入，体验上统一，数据和权限上与普通记忆完全隔离。

### 12.1 分区

- 凭据库：密码、密钥、Token、Cookie、私钥等可用于执行操作的秘密。
- 受保护个人资料库：身份证、详细地址、联系方式等高敏感个人资料。

### 12.2 星图关联

普通星图只显示脱敏关联，例如“已关联一项受保护 SSH 凭据”。不显示明文、完整地址、秘密值、敏感文件路径或可直接利用的细节。

### 12.3 管理与授权

- 只有用户可手动新增、查看、修改、替换、禁用和删除完整敏感条目。
- Hermes 默认只能知道存在脱敏关联，无权读取或自动调用完整值。
- 用户可授予 Hermes 一次性、明确目标、明确操作、最小范围、可过期且可撤销的使用权限。
- 推荐的授权语义是“允许完成某项操作”，而不是把秘密明文注入模型上下文。
- 所有访问、授权、撤销和使用都必须有审计记录。
- 敏感值不得进入普通记忆、星图搜索、整理报告、派生摘要或模型上下文。

## 13. 模型与数据处理要求

- 支持外部 API、本地部署模型及自定义 base URL。
- 模型选择可按任务配置，例如事实提取、深度整理、摘要和检索辅助可分别配置。
- 在调用外部 API 前，应尽量最小化发送内容，并保持可配置的数据边界。
- 高敏感数据、凭据明文和受保护个人资料默认不得发送给任何模型。
- 外部模型不可用时，原始证据采集、治理、浏览、手动编辑和基础检索不得丢失数据；自动模型整理可延迟重试。

## 14. 可靠性、安全性与验收

### 14.1 必要质量要求

- 原始证据完整、只读、可追溯；敏感值例外，必须脱敏或隔离。
- 任何后台整理失败、重试或重复执行都不得丢失证据或制造重复稳定记忆。
- 更正、脱离、忘记和永久清除必须传播到受影响的派生内容与星图。
- 数据必须可备份、恢复、迁移和核验；永久清除应有明确不可恢复语义。
- 用户编辑与自动整理的历史必须可审计。

### 14.2 V1 验收标准

1. 每条正常召回的事实、情节和长期脉络均可回溯至原始证据。
2. 无足够证据的模型推测不会自动成为稳定事实。
3. 同一事实在重复整理后不会产生多个稳定副本。
4. 用户更正后，旧版本不会被默认召回，相关派生内容会更新。
5. 用户永久清除后，相关证据、派生记忆、索引和星图关联均不再出现。
6. Hermes 多 profile/实例可共享指定的短期和长期记忆。
7. 星图支持浏览、追溯、更正、脱离、忘记、永久清除及实体合并/拆分。
8. 凭据与受保护资料不出现在普通检索、摘要、星图节点正文或模型上下文中。
9. 用户可通过星图管理受保护资源，并可查看授权和使用审计。
10. 系统能生成并在星图中展示默认周期为 7 天的整理报告。
11. 状态模块在相同初始状态、事件与时间输入下产生可复现的状态快照与建议。
12. 状态模块不会自行写入事实记忆、读取敏感明文、使用凭据或触发外部操作。
13. Hermes 可通过稳定接口完成召回、浏览、来源追溯、当前状态更新和记忆更正。
14. 每次召回都能说明候选命中通道、来源证据和最终排序依据。

## 15. 连续互动状态模块（jiwen 理念整合）

本模块借鉴 [jiwen](https://github.com/ClaraShafiq/jiwen) 的核心理念：用连续数值随时间漂移和阈值判断来产生确定性状态/建议，而非依赖随机触发。它是 Hermes 记忆系统中的独立子域，不是事实层、情节层或长期脉络层的一部分。

### 15.1 V1 定位与边界

- 模块纳入 V1，默认启用状态记录和状态面板；状态驱动的主动消息默认关闭。
- 状态用于形成可读的互动上下文和行为建议，例如“适合简洁回应”“应继续专注当前任务”“可以提醒用户查看整理报告”。
- 状态不能创建、删除、更正或提升任何事实记忆；状态变化不是关于用户或世界的事实。
- 状态不能读取敏感明文、绕过凭据授权、改变用户权限或直接执行工具调用。
- 长期脉络只能以带来源和上限的“参考信号”影响状态；不得让未经证据支持的叙事直接改变关键状态。

### 15.2 默认状态维度

jiwen 的五轴是特定角色设计。为适配 Hermes，V1 使用语义中立、可按 profile 配置的默认维度：

| V1 维度 | 参考 jiwen 概念 | 作用 |
| --- | --- | --- |
| 互动需求 | connection | 当前是否有必要与用户继续互动或提醒 |
| 表达克制 | pride | 主动性与克制之间的平衡，不预设人格含义 |
| 情感效价 | valence | 当前互动语气偏积极、中性或低落 |
| 激活度 | arousal | 平静、专注、紧迫或焦躁程度 |
| 任务沉浸 | immersion | Hermes 是否处于已声明的任务/活动中 |

每个维度的名称、范围、初始值、漂移规则、阈值、是否启用和 profile 覆盖策略均须可配置。默认状态归属于同一个 Hermes 身份；profile 可提供不同的语气映射，但不得制造相互矛盾的独立事实。

### 15.3 输入与更新规则

状态变化仅可来自以下输入：

1. 实际经过时间产生的确定性漂移；系统重启或延迟后必须按真实经过时间补算。
2. 已发生对话经受控分析后产生的结构化状态增量；分析结果必须标记来源、置信度和可撤销性。
3. 用户或 Hermes 明确设置的活动、任务状态和用户可用状态。
4. 来自已验证事实、情节或长期脉络的有限参考信号，且可展开查看其证据。

模型不可直接写入任意状态值；模型只能提交受校验的候选增量。模型不可用时，时间漂移、状态查看、历史追溯和手动调整仍可正常工作。

### 15.4 输出与安全门控

状态模块输出以下受控事件；这些事件均为建议或观察，不是已执行动作：

| 事件 | 含义 | 默认效果 |
| --- | --- | --- |
| `state_snapshot` | 可审计的当前状态和变化记录 | 写入状态历史 |
| `observe` | 注意到沉默、状态变化或任务持续等情况 | 仅在状态面板显示 |
| `tone_guidance` | 面向 Hermes 的简短语气/表达建议 | 可作为最小化上下文指导 |
| `suggest_contact` | 建议在合适渠道恢复互动或提醒用户 | 默认不外发 |
| `suggest_activity` | 建议继续、切换或结束已声明活动 | 默认不执行工具调用 |

默认禁止状态模块直接产生外发消息、通知、工具调用、敏感信息访问或凭据使用。只有用户在星图中显式启用某类动作，并为具体渠道/范围授予权限后，上层 Hermes 才能将建议转换为行动；每次行动仍须进入审计记录。

输出给语言模型时必须是最小化的自然语言指导，而不是原始数值、完整状态历史或敏感来源。状态本身不得作为“用户事实”写回普通记忆。

### 15.5 可解释性、模拟与治理

- 星图的状态面板必须显示状态摘要、近期变化、输入来源、阈值触发和建议历史。
- 用户可查看、暂停、重置或调整状态配置；手动调整应记录为用户操作。
- 修改状态参数前，应支持使用历史事件线进行模拟，比较不同参数下的状态轨迹和触发结果。
- 相同初始状态、输入事件和时间间隔必须得到一致结果；缺失输入或模型分析失败时不应产生自主动作。
- 状态记录应具有独立生命周期和审计历史，不与普通事实遗忘规则混淆。

## 16. 后续阶段预留

- 多用户、多 Agent 和细粒度授权模型。
- Hermes 通知渠道中的整理报告提醒。
- 状态建议到主动行为的更丰富映射（需单独的用户授权、渠道策略和安全评估）。
- 云端同步、外部知识源和通用 Agent 接入。
