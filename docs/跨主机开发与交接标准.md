# 跨主机开发与交接标准

从 2026-07-21 起，每个开发批次都必须停在“另一台主机只通过 Git 即可继续开发”的状态。

## 提交完成定义

- 代码、迁移、Compose、脚本、测试、文档和公开示例配置全部已跟踪；
- 私有 env、Vault key、数据库、备份和真实 Hermes 导出必须忽略；
- 模型 key 只位于仓库外 `0600` secret 文件，不出现在 env、命令行、日志、状态文件或交接文档；
- `VERSION`、Python 包版本、前端版本和镜像标签一致；
- `docs/handoff.md` 记录当前版本、完成项、未完成项、运行边界和下一步；
- 生产运行版本必须有只读 deployment bundle，来源角色必须有 `SOURCE-POLICY.json`；二者都属于运行资产，不提交 Git；
- 工作树干净，当前 HEAD 等于 upstream；
- `scripts/handoff-check.sh` 通过；涉及容器/数据的变更另外通过隔离 release Gate。

`handoff-check.sh` 包含锁文件、脚本语法、Ruff、单元测试、前端 typecheck/build、必要文件跟踪和 upstream 一致性检查。

## 新主机接手流程

```bash
git clone https://github.com/evergardener/agent-memory.git
cd agent-memory
git switch main
uv sync --frozen --extra dev --extra migrations
npm --prefix frontend ci
bash scripts/handoff-check.sh
```

如果只继续开发，不需要也不应获取生产密钥和真实数据。需要接管生产运行时，按生产候选手册的迁机流程单独授权和传输 env/state、来源策略、deployment bundle、已验证备份和独立 Vault key。源码 checkout 不等于已获准管理运行容器。

容器更新、重建或运行版本重新绑定必须在隔离 release Gate、升级前备份恢复和用户确认后执行；完成后必须复验 API/Worker/迁移、来源 inventory、星图、备份恢复和真实 Hermes 业务。不得把“可从另一台主机拉取源码”误解为“可复制或启动生产运行态”。
