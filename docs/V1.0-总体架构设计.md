# Agent Memory for Hermes — V1.0 总体架构设计

> 2026-07-20 实施修订：`1.0.0-rc.6` 是阶段 A/B 的最近不可变候选；当前工作树已完成阶段 C 的类型化关系、可重叠社区、治理、布局和主/子宇宙隔离验证，尚未获准启用正式空间或发布下一候选版本。

## 1. 目的与设计边界

本文将 [`V1.0-项目需求文档.md`](V1.0-项目需求文档.md)、[`V1.0-星图概念与投影设计.md`](V1.0-星图概念与投影设计.md) 和 [`技术选型方案-V1.0.md`](技术选型方案-V1.0.md) 落实为服务边界、数据流和运行规则。它不定义具体 ORM、UI 组件或模型名称；这些必须服从本架构中的数据与安全约束。

V1 是本地 Docker 部署的单用户、单 Hermes 记忆服务。多个 Hermes profile/实例属于同一个稳定 `shared_namespace`，但每份输入和每次操作均保留来源 profile、会话和关联 ID。Vault 是同一产品内的受保护子域，不是普通记忆库。

## 2. 逻辑架构

```text
Hermes profile / instance
        │  MemoryProvider（Python）
        │  仅发送受控事件，读取受预算限制的上下文
        ▼
Agent Memory API ───────────────────── 星图与 Vault Web UI
        │       │                              │
        │       └────────── OpenAPI ───────────┘
        │
        ├── PostgreSQL + pgvector
        │     evidence / memory / retrieval / state / vault / ops / audit
        │
        └── durable job enqueue
                    ▼
            Worker（Python） ─── Model adapter ─── 本地或外部模型
```

部署服务固定为 `postgres`、`api`、`worker`。API 仅绑定宿主 `127.0.0.1`；PostgreSQL 与 worker 不暴露宿主端口。前端构建产物由 API 提供。模型服务是外部可配置依赖，不构成 Compose 的强制服务。

## 3. 组件职责与禁止事项

| 组件 | 必须负责 | 不得负责 |
| --- | --- | --- |
| Hermes Provider | 生命周期适配、回合幂等提交、召回注入、工具路由、失败降级 | 保存业务数据、直连数据库、解密 Vault、执行深度整理 |
| API | 鉴权、输入校验、事务写入、权限/作用域裁决、UI 与 Provider 契约、审计 | 在 HTTP 请求内执行长时间模型调用或无恢复保证的后台任务 |
| Worker | 脱敏、抽取、索引、整理、重建、报告、状态计算、可恢复任务执行 | 绕过 API 的安全策略、把敏感明文送入模型 |
| PostgreSQL | 唯一持久事实源、事务、检索投影、任务和审计 | 保存根密钥或替代受控备份 |
| Web UI | 浏览、追溯、纠正、主体/星系治理、Vault 的人工操作入口 | 将全量记忆或 Vault 明文加载到浏览器图元素；把事实分类误作实体硬分区 |

## 4. 核心数据流

### 4.1 每回合路径

1. Hermes 在 `on_turn_start`/`prefetch` 向 API 发送查询、profile、会话和 `shared_namespace`。
2. API 先作 namespace、scope、状态和敏感策略过滤，再执行词法、语义、关系候选融合，返回带预算、来源 ID 和简短理由的上下文。
3. Hermes 完成对话或工具调用后，由 `sync_turn` 提交规范化的用户消息、助手消息、工具结果与关联 ID。
4. API 在一个事务中写入只读证据事件、脱敏投影、审计行和带幂等键的处理任务；提交成功即返回，不等待抽取。
5. Worker 处理证据，产生候选事实、实体关联和可检索投影。主体映射来自受控 profile 配置，不能由模型按名称猜测；模型失败仅使派生任务重试，绝不丢失证据。

### 4.2 整理与治理路径

即时任务只做规则脱敏、基本归类和必要索引。轻量整理可关联实体、发现重复、过期当前状态。深度整理在空闲、积压、周期或用户手动触发时产生 episode、arc 和整理报告。

自动操作必须写入 `audit`，并以来源证据和输入版本为依据。存在冲突、证据不足或敏感边界模糊时，停留在候选/待确认状态，而不能生成确定性事实。

### 4.3 Vault 路径

用户只通过 Vault UI 创建、查看、修改或删除受保护资源。普通事件进入时仅检测并生成脱敏占位与关联，不生成可召回明文。Hermes 请求使用资源时，API 必须检查用户创建的、未过期的最小能力授权；默认由 API/worker 代理操作并返回脱敏结果。明文披露为单独的、默认关闭且强审计的人工动作。

### 4.4 星图投影路径

星图不是另一套记忆真相，而是从主体、canonical entity、当前有效事实和实体关系生成的可治理投影：

基础 `planetary-v2` 仍由 API 查询时生成，天体集合只有 Subject 与 entity；事实、episode、arc 和 Vault 作为独立 overlay 返回。阶段 C 在此基础上增加 `community-projection-v1`：类型化关系与关系事实保存在 `memory`，可重建星系与成员解释保存在 `projection`，人工治理和布局独立持久化。星系从不成为事实或实体的唯一硬归属。

1. Provider 的 `source_profile`/实例先映射到稳定 profile 人格主体；同一 profile 多实例不得生成多个主体。
2. 用户主体和 profile 人格引用 canonical entity，在主宇宙投影为恒星；普通实体投影为行星。
3. 自动社区只以有证据支持的行星—行星关系为主要输入。主体—行星关系用于视角和排序，不能单独形成星系。
4. 社区结果写入可重建的星系与多对多成员投影；人工命名、固定、排除和布局偏好作为独立、可审计的治理层覆盖自动布局。
5. 长期、阶段、当前、生命周期和敏感性通过查询镜片改变亮度/范围，不写成唯一 `galaxy_id`。
6. episode/arc 按需投影为临时星座和星流；关闭播放后不改变 canonical entity 或星系成员关系。
7. 更正、脱离、忘记、清除或实体合并/拆分触发受影响社区、星座和检索投影重建。

社区算法只能生成布局和候选解释，不能创建实体关系或稳定事实。任何模型提出的星系名称、摘要或成员调整必须标记为派生建议，并服从用户固定规则。

## 5. 召回控制面

召回不是对数据库的无过滤搜索。每一请求按下列顺序裁决：

1. 验证本地服务令牌、`shared_namespace` 与请求来源；
2. 普通 `conversation` 召回剔除 forgotten、永久清除、孤立和过期记录；只有显式 `explicit` 检索允许 forgotten 以词法/实体强匹配候选出现；
3. 应用 scope、用户治理状态与敏感策略；
4. 并行取得词法、语义和关系候选；
5. 在应用层执行 RRF 与时间、证据强度、状态匹配、访问预算重排；
6. 截断为 Provider 的 token/条数预算，返回来源可追溯 ID；
7. 写入不含正文的召回审计。

模型不可用时，词法和关系检索仍可用；向量索引延迟时不得阻断回合。episode/arc 与事实使用同一融合器，但返回其底层 evidence ID。API 不返回 Vault 明文，也不把 isolated/purged 内容作为提示线索。

## 6. 可靠性规则

### 6.1 写入与幂等

Provider 为每个 Hermes 回合产生稳定 `idempotency_key`（namespace、会话、turn、事件类型的确定性组合）。API 在数据库唯一约束下去重。任何任务均携带 `input_version` 与幂等键；同一输入的重复执行不可产生重复事实、边或索引文档。

### 6.2 失败分级

| 失败 | 回合行为 | 后续行为 |
| --- | --- | --- |
| API 暂不可用 | Hermes 继续工作，不注入记忆或使用上次安全缓存 | Provider 指数退避并记录本地诊断，不缓存敏感正文 |
| PostgreSQL/写入失败 | 不确认事件已保存 | Provider 可安全重试同一幂等事件 |
| 模型失败 | 证据写入成功，派生层不更新 | worker 有限重试，最终转人工处理/下次周期重试 |
| 索引失败 | 不影响证据或事实 | 标记投影待重建；词法/关系路径降级 |
| Vault 授权失败 | 不返回明文、不执行目标操作 | 记录拒绝审计，提示用户在 Vault 明确授权 |

### 6.3 恢复原则

派生表、向量和自动星图投影均可从未被清除的证据和事实重建；主体/profile 人工映射、星系人工命名/固定/排除和布局偏好必须作为治理数据单独备份恢复。原始证据不能被派生流程修改。恢复演练必须覆盖空实例恢复、索引重建、未完成任务接续、星图治理恢复与 Vault 条目解密验证。

## 7. 安全边界

- 根密钥不进入数据库、日志、Compose 文件或普通 `.env`；由受限挂载/secret 提供。
- 外部模型仅获得经策略裁剪、脱敏的最小文本；模型请求与响应日志不得保留敏感正文。
- API 使用安装生成的本地令牌；UI 使用独立登录会话与 CSRF 防护。令牌轮换使旧 Provider 失效但不删除数据。
- 审计采用追加写入。用户数据的永久清除按批准的清除工作流执行；审计仅保留法律/运行所需的最小无正文记录。

## 8. 架构验收

上述基础 Spike 与星图专项 Gate 均已通过。阶段 C 当前证据包括：同 profile 多实例稳定映射、唯一行星、3 个真实关系星系/9 个成员、重叠 fixture、零单体负例、镜片不改变成员、治理重建保留、Hermes Provider 召回以及空库恢复。正式空间启用仍以用户验收和单独授权为前置条件。
